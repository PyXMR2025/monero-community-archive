## vtnerd | 2017-09-15T00:59:10+00:00
I just recalled more of my previous design rationale. The `thread_group` had an internal `boost::optional` so that the `mutex` / `condition_variable`, etc., did not have to be initialized if the thread count == 0 (they were often being created temporarily on the stack, unfortunately). The `task_region` had a `shared_ptr` in part to delay the creation of the `mutex` / `condition_variable` pair too. Why I didn't go with the `boost::optional` for `task_region` instead of the allocation I don't remember, other than it looks less "sketchy".

Anyway, a `boost::optional` inside of the `waiter` type is an easy method to delay construction of those synchronization types if the implementation frequently runs on the local thread due to hitting the `max` queue. I just wanted to point this out in case you noticed this while running perf tests / profiling.
