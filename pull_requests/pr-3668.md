# Original Description
Resolves https://hackerone.com/reports/340012 (public disclosure very soon).

@fluffypony @danrmiller We need to remove miniupnp from the **release** build system completely because:
1. We shouldn't trust CMake to be reliable (it's CMake)
2. Patches are needed to ensure system headers aren't used and, if they aren't reliable then see point 1.

To remove, on Ubuntu for example, you would need to remove both libminiupnpc-dev and miniupnpc packages.

Once @fluffypony sets up a monero-project miniupnp (like we did with unbound), I can push [branch monero](https://github.com/anonimal/miniupnp/tree/monero) from my fork which includes all the necessary patches and then update this PR.

Important note: I would even advise to completely remove system miniupnp from all buildbot machines after we adjust our CMake to build our submodule/in-tree exclusively. There's no legitimate reason (that I can think of) to continue to depend on a system's miniupnp.
