## nahuhh | 2025-07-10T13:20:40+00:00
- By default the estimations target 10mb batches
- estimates are conservative, but can still be exceeded by ~100% under spike volume
- current serialization limits break syncing if you exceed ~25mb in a batch
- until serialization limits are extended / removed, 10mb is the largest we can safely default to. Any larger has potential to hit serialization limits
- if a single block is expected to exceed 1/2 of the batch size target, then syncing more than 1 block would exceed the target, hence the "5mb"
- the max this can be set to, is 50mb (1/2 of the levin limit to add safe-zone for underestimation due to spikes in volume). 50mb cant be used today though, because of serialization 


Also to note
- downloading of 10mb batches results in recalculating the estimation more often, and more reliable downloads (since monero likes to ban peers who dont complete the upload fast enough)

## nahuhh | 2025-08-20T22:08:00+00:00
> I wonder why we are targeting the specific short-term limit, if we don't use the rule that the next block must be less than 2x the weight.

since we arent syncing 1 block at a time (the block creation size limit is set 1 block at a time), the median of the last 100 blocks that we've synced might not be indicative of what we are syncing. I also feel that 2x the max would be too aggressive and you'd almost always pull undersized batches.

> This encodes the assumption that future blocks *probably* won't eat into the block penalty. Is this a good assumption? 

We allow for the incoming batch to exceed the target by 100%, in bursts. If the max synced block size within the window was 2mb, then we start getting a flood of 4mb txs, the next batch will be 20mb. the following batches will be 10mb. The max that batch-max-weight can be set to is 50mb, to account for the packet size limit (100mb). Its more of a target (a soft limit), than a hard one.

note: with serialization, you can't really use the full 50mb. ~12mb is the effective max (25mb bursts). 

In practice, this works quite well imo - using the max of the past 100 blocks to guesstimate the incoming block sizes. Also, since we download way ahead, we are also comparing to avg block size for each span in the queue
