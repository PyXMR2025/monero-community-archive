# Original Description
- Fixed uninitialized `state->x` warning
- Fixed broken code with `-O3` or `-Ofast`

The old code is known to break GCC 10.1 and GCC 11.4

Closes #9041

# Discussion History
## jeffro256 | 2023-10-29T22:44:58+00:00
This is technically undefined behavior according to the strict aliasing rule. 

## SChernykh | 2023-10-30T09:04:00+00:00
Casting `uint8_t` buffer to `uint64_t` is technically UB, yes. But since we only XOR bytes here, it will not be affected by endianness. Compiling with `-fsanitize=undefined` doesn't find anything in this code. I will also add a proper data alignment for `buffer` to be sure we can access it as `uint64_t` without doing unaligned accesses.

## jeffro256 | 2023-11-04T00:15:49+00:00
Even if it is correctly aligned, there is no guarantee by the language that cached, deferenced bytes will actually reflect the contents of `state->buffer` since the aliases are of non-compatible types. If we insert a `memcpy`, we can guarantee portability, and most compilers will optimize the superfluous instruction out anyways. I haven't looked into it, but the line `state->x[(8+i) >> 1][(8+i) & 1] ^= ((uint64*)state->buffer)[i];` could very well be one of the reasons that the old code broke on different optimization levels.

So instead of:
```
uint64_t* x = (uint64_t*)state->x;
const uint64_t* buf = (uint64*)state->buffer;

for (int i = 0; i < 8; ++i) x[i] ^= buf[i];

E8(state);

for (int i = 0; i < 8; ++i) x[i + 8] ^= buf[i];
```

I say we do:
```
uint64* x = state->x;

for (i = 0; i < 8; i++) 
{
    uint64 b;
    memcpy(&b, &state->buffer[i << 3], sizeof(b));
    x[i] ^= b;
}

E8(state);

for (i = 0; i < 8; i++) 
{
    uint64 b;
    memcpy(&b, &state->buffer[i << 3], sizeof(b));
    x[i + 8] ^= b;
}
```

It's a lot wordier, but it complies with the C standard.
