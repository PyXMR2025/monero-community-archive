# Original Description
Edge case identified by @jeffro256 

If you create a wallet with legacy software pointing to an untrusted daemon and that legacy wallet has 0 receives, and then you re-open the wallet using FCMP++ compatible software again pointing to an untrusted daemon, then the wallet will granularize the starting refresh height to a height further back (to avoid revealing a fingerprint to the daemon [here](https://github.com/seraphis-migration/monero/blob/dbe95f3ba275ed1f84bf1263f617594cf013b958/src/wallet/wallet2.cpp#L3402-L3411)).

However, `m_blockchain` will then be synced "further ahead" of the start height requested of the daemon, and so when the wallet attempts to initialize the tree, [this fails](https://github.com/seraphis-migration/monero/blob/dbe95f3ba275ed1f84bf1263f617594cf013b958/src/wallet/wallet2.cpp#L3885).

The sync flow expects `m_blockchain` and `m_tree_cache` to maintain parity. This PR's solution is to detect this specific edge case, and sync the tree up to `m_blockchain` when initializing the tree. This guarantees the tree is at parity with the wallet's sync state.
