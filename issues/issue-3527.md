# Original Description
I generated a 25-word mnemonic seed with [xmr.llcoins.net's Cryptonote Address Tests tool](https://github.com/luigi1111/xmr.llcoins.net/blob/master/site/addresstests.html) by using a custom hexadecimal seed (inserted into box no. 2 and pressed `Gen 1.`). Importing this certain mnemonic into Monero Core via the GUI changed almost all of the words in the mnemonic by the time I got to the last step of the import wizard, and this changed version was stored by the wallet as well. Thankfully, the public address is the same as the one I can derive in the tool.

I can't replicate the behavior with other mnemonics generated by the tool. (I tried with ones generated from other custom hexadecimal seeds, and also with random mnemonics generated with the `Random` button next to box no. 1.)

Why did this happen?

Paging @luigi1111 as it may be related to their code.

## moneromooo-monero | 2018-03-31T17:47:11+00:00
The wallet code indeed does not store the seed. It is recreated whenever needed from the secret spend key. Mnemonic seeds translate to a key value that's less than l, by construction. This also means not all 25 word lists are valid seeds (though they'll be transformed to a valid one in the way I suspect yours was).  As far as I know, whether your original 256 bit random value is lower than l or not has no (unreasonable) influence on the security of the key modulo l. Some keys are more likely to be created than others due to the modulo operation, but I believe that is negligible for security. luigi will know way more about this though.
