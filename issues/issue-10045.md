# Original Description
Bad [memory alignment](https://blog.quarkslab.com/unaligned-accesses-in-cc-what-why-and-solutions-to-do-it-properly.html) can cause slowdowns, program termination, and is generally undefined behavior. LMDB guarantees a 2-byte alignment on nodes: http://www.lmdb.tech/doc/group__internal.html. However, some objects have a greater alignment than 2 (most notably 32-bit and 64-bit integers). This wouldn't necessarily cause alignment issues as long as objects with alignment greater than 2 are copied into a memory segment with correct alignment. Unfortunately, this is not the case for a lot of the `BlockchainLMDB` code. And this isn't just a theoretical issue either. If you apply the following git diff, compile, and run as a debug build the node, it will fail after an alternate block is handled to the core.

```diff

index ff44923ff..18c80e8e5 100644
--- a/src/blockchain_db/lmdb/db_lmdb.cpp
+++ b/src/blockchain_db/lmdb/db_lmdb.cpp
@@ -136,6 +136,11 @@ private:
   std::unique_ptr<char[]> data;

+  static constexpr std::size_t align = alignof(std::remove_reference_t<decltype(*(p))>); \
+  assert(reinterpret_cast<std::uintptr_t>(p) % align == 0);                              \
+} while (0);
+
 }

   const alt_block_data_t *ptr = (const alt_block_data_t*)v.mv_data;
+  assert_ptr_alignment(ptr);
   if (data)
     *data = *ptr;
   if (blob)
```

This demonstrates that the alignment for LMDB key-value pairs storing `alt_block_data_t` entries is less than the actual alignment for `alt_block_data_t`. My hypothesis why LMDB misaligns these entries and not others is that in `BlockchainLMDB::add_alt_block()`, the `alt_block_data_t` is first copied into a `char` array (alignment of 1) before being passed to LMDB. LMDB might happen to preserve the alignment otherwise. On x86 systems, unaligned accesses are allowed, but are just very, very slow. However, for example, on ARM systems this could trigger a bus error and terminate the program. There is also the discussion of whether this breaks the strict aliasing rule. 

The fix for this is to explicitly `memcpy()` values of alignment greater than 2 when we need to load them from LMDB, which will be a very large refactor.

You can also run the extended alignment enforcement patch here: 733774c3e0ff07570713b32cfadd34506814486e. It would be interesting to see which alignment failures occur where on which systems.
