I don't understand the first check. `total_size` already has the size of the current transaction. Is the goal to roughly estimate what the size of the next transaction will be ?

Also a move iterator (which does a `std::move` on `operator*` calls) allows for efficiency in "both"cases:

```c++
std::size_t first = 0;
std::size_t total_size = 0;
for (std::size_t i = 0; i < txes.size(); ++i)
{
  const std::size_t bytes = txes[i].size();
  total_size += bytes;
  if (total_size + bytes / 2 > MAX_TXES_BYTES_IN_TX_NOTIFICATION || i == txes.size() - 1)
  {
    if (i == txes.size() - 1 && first == 0)
    {
      new_txes.txs = std::move(txes);
      txes.clear();
    }
    else
    {
      new_txes.txs.assign(
        std::make_move_iterator(txes.begin() + first), std::make_move_iterator(txes.begin() + i + 1)
      );
    }
    total_size = 0;
    first = i + 1;
    post_notify<NOTIFY_NEW_TRANSACTIONS>(new_txes, context);
  }
}
```
A `reserve` (just) before the `assign` would guarantee that multiple calls into `assign` only allocate the first time. I would avoid doing the `reserve` before the loop, because otherwise its allocating for no reason pretty much all the time.
