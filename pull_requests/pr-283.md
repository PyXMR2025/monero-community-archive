## jeffro256 | 2026-01-31T01:12:26+00:00
Why packing?

## j-berman | 2026-01-31T02:42:44+00:00
It's going in the db, so just as a precaution to prevent any alignment issues that may cause the struct to be larger than the sum of its members

## jeffro256 | 2026-02-01T05:52:45+00:00
With C++17, I think that a more portable / comprehensive option would be to layout the struct in a sensible manner (as already done), drop the `#pragma pack` and add an assert statement `static_assert(std::has_unique_object_representations_v<UnifiedOutput>);` This ensures you don't butcher alignment by packing (which can cause segfaults on ARM), and also catches trap representations on really obscure architectures. 

## j-berman | 2026-02-03T00:21:26+00:00
In the latest I decided to serialize the `UnifiedOutput` before storing it, to ensure it's stored in a consistent portable way (the memory layout of `std::variant` is apparently not necessarily portable across platforms). Serializing removes the need for both packing and that static assert on the struct, while also solving portability. It adds a little complexity, but seems a stronger impl this way ensuring the object is stored the way we think it is.

Of note, the packing was actually necessary for this struct to be 73 bytes (there's 8 bytes for the `unified_id`, so it's aligned to 8 bytes, and there's 1 extra byte for the variant tag.. so without packing, the `UnifiedOutput` is actually 80 bytes). And this also fails `static_assert(std::has_unique_object_representations_v<UnifiedOutput>)`.

## jeffro256 | 2026-02-08T21:02:29+00:00
```suggestion
    VARINT_FIELD_F(unified_id)
```

## j-berman | 2026-02-10T01:01:49+00:00
If this is a varint, the `get_outs_at_last_locked_block_idx` db reads would be less efficient because we wouldn't be able to use `MDB_NEXT_MULTIPLE` to get a page of items from the `locked_outputs` table at a time ([that requires all items in the table have a fixed size](http://www.lmdb.tech/doc/group__mdb.html#ga1206b2af8b95e7f6b0ef6b28708c9127)).

The same principle is applicable to the `mdb_leaf` elem when reading the `leaves` table in `find_leaf_idx_by_unified_id_bounded_search`.

The choice to use a `uint64_t` here I think is a reasonable tradeoff to take
