SHA1 is too close to bruteforceable

# Discussion History
## omartijn | 2019-04-26T06:55:24+00:00
For added clarity we could use the `SHA256_DIGEST_LENGTH` constant here, as well as the other locations where 32 is hard-coded.

## moneromooo-monero | 2019-04-26T09:15:12+00:00
I guess, but it's SHA256, it's already in the name (and it makes the error messages have to be constructed at runtime which is a pain) :)

## omartijn | 2019-04-26T09:43:52+00:00
Sure, the fact that it's SHA256 does make it sort of obvious why it's 32 bytes. Personally I like to be verbose in these situations, if only because it makes grepping easier.

Messages don't necessarily have to be constructed at runtime though. With some preprocessor magic you can turn this numeric constant into a string literal, which you could then use. Something like this:

```
#define STR_HELPER(x) #x
#define STR(x) STR_HELPER(x)
```

You could then make your error message like this, which makes this a compile-time string literal:
`MERROR("SHA-256 fingerprint should be " STR(SHA256_DIGEST_LENGTH) " bytes long");`

I do wonder whether that's not taking it too far, though. The benefit I see of this is that if we would want to change the hashing algorithm for fingerprints yet again (say we want to go to sha384) it makes it easier to find all the places to change.
