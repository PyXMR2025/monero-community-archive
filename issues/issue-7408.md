# Original Description
Opening this issue to discuss and track a potential addition of a gRPC wallet interface.

Some discussion has already happened on [on reddit](https://old.reddit.com/r/Monero/comments/l6fxkl/gauging_interest_grpc_api/) and IRC/matrix.

The basis for this would be to unify the functionality of unary RPC calls and pub-sub ZMQ messages into a single interface. The interface proposed is minimal, as the scope of it would be to ease integration but not replace the JSON-RPC or ZMQ APIs.

I believe gRPC would achieve this by: 

- Provide a single dependency and framework to work with for synchronous and asynchronous calls.
- Enable more granular payment notifications.
- Possibility to combine documentation to the interface definition.
- Extended wallet client library support to all languages supported by gRPC.
- Autogenerated documentation.

In terms of risks:

- New dependency on the gRPC plugin for the protoc compiler (protobuf already being a dependency for trezor support).
- Reliance on Google maintaining the library/plugin.

Potential questions that may arise:

- Authentication can be done via [TLS](https://grpc.io/docs/guides/auth/)

Submitted  #7409 for initial feedback before going further.

This is another serialization format _and_ rpc layer. I think the negatives outweigh the benefits.

* The scheme is less supported in other systems than HTTP-JSON-RPC
* `protobuf` is one of the most irritating formats - its not self-describing and the output of objects/structs (basically) has to be buffered to write the variable-length size field before the object. Whereas the ZeroMQ code already "streams" outputs to JSON, and the HTTP (epee) code could do the same with some (slightly painful) refactoring. This has a big impact on performance because many binary values have to be sent for wallet processing.
* `protobuf` injects its auto-generated structures into codebases, so now there's a third copy (ZeroMQ and HTTP already could use a good merging). I'm closer to providing a "slice" concept to the epee serialization code so that even when copies are necessary its just a ref count change. The `protobuf` code enforces `std::string` in both directions in the auto-generated code. FWIW ZeroMQ code, which uses rapidjson for reading in, probably has this limitation with its DOM (assuming it can't be templated to something else).


> Provide a single dependency and framework to work with for synchronous and asynchronous calls.

You'd have to expand on this. ZeroMQ can do all of the stuff gRPC is doing at the protocol layer (afaik - see ROUTER/DEAER sockets), but gRPC might have built-in mechanisms for async callbacks in the language of choice.


> Possibility to combine documentation to the interface definition.

This is possible in our current approach too, but credit to gRPC for already written code.

> Enable more granular payment notifications.

I'm not sure what you mean. `monerod` does not handle anything wallet related. Most of the wallets are built around `wallet` which manages one wallet at a time. ZeroMQ could be added to the wallets but has not. 

The subscription topics for PUSH events are strings and the server side can read them in a `XPUB` socket. So the PUSH "topics" do not have to be fixed in advance. The work on the client side is generating the "topic" string - `json-wallet-watch-ADDRESS`, etc. A little funky, but async and basic work for clients to implement. The client just has to pass that string to a ZeroMQ function in their language.

> Extended wallet client library support to all languages supported by gRPC.

ZeroMQ is already supported by many languages, does the bulk of the work for RPC and PUSH calls for clients, and is agnostic to the internal format. We use JSON which is widely supported. Switching to gRPC means it auto-generates code structures that then have to be glued to the rest of the system - there's no realistic time savings generally. You either map a DOM or map this custom protobuf thing.

> Autogenerated documentation.

The auto-documenting feature presumably only lists field names and their types. Some (actually not terrible) modifications to the epee code should allow that without an additional format+rpc dependency. Although a benefit to gRPC because it's already written code.

Unfortunately, optional build targets have a tendency to become mandatory: If the optional target has dependencies, and developers have these dependencies installed on the build machine (which they likely will, so as to test compatibility with the optional target), then the non-optional targets will compile, even if they're broken without said "optional" dependencies. So, you end up having to run two sets of CI builds, with and without the "optional" components, to confirm the optional components really are optional. With the JSON based RPC, there's already no mechanism to do this. You have to hope that your 3 years out of data client library is compatible with the current version of monero-wallet-rpc. GRPC allows you to set a schema version compatibility guarantee on the server. 

If you add a field or new endpoint in the gRPC server, existing generated clients will continue to work! - just like the current JSON RPC system. 

With gRPC we get the same backwards compatibility as the current JSON RPC, but we also get always-up-to-date client libraries for new fields and methods. No more chasing down the author of a library that was last updated years ago to add some new feature added to the RPC server.
