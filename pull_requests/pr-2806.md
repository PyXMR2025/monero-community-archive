## moneromooo-monero | 2017-11-14T08:59:00+00:00
Is that different from boost::optional<uint32_t> subaddr_account = boost::none; ?

## binaryFate | 2017-11-14T09:11:50+00:00
"boost::optional<uint32_t> subaddr_account = boost::none;" does not silence the warning.   
   
I think it's because it assigns a "normal" value but leaves the boost::optional specific part uninitialized (which is the one taken into account with a boolean check `if (subaddr_account)` as used further below). boost::optional variables default to false for this test, but GCC doesn't take it into account. This can only be explicitly initialized by a value returned by a function, thus the lambda. I'm not sure how to explain better, and I'm also not super familiar with it.

See https://stackoverflow.com/questions/21755206/how-to-get-around-gcc-void-b-4-may-be-used-uninitialized-in-this-funct  

## binaryFate | 2017-11-14T09:25:06+00:00
In other words, setting subaddr_account = boost::none would work for testing `if (*subaddr_account)` but not `if (subaddr_account)`.

## moneromooo-monero | 2017-11-14T09:42:53+00:00
OK. That page has a simpler workaround (boost::make_optional) fwiw.

## binaryFate | 2017-11-14T09:49:21+00:00
I did not want to use that other workaround because IIUC it lets the compiler define the type from a value only, and I am not sure how to cleanly make sure it picks `uint32_t`. While with the workaround in this patch, despite the "auto" you can explicitly tell the compiler in that same line that it must be an uint32_t.
