# Original Description
If you send a Monero tx, it enters the pool, and then restore the wallet, then the wallet won't identify the spend tx in the pool. If a user then attempts to construct *another* tx, the wallet may attempt to re-spend an output already spent in the pool, causing the node to reject the tx as a double spend. Note: this bug requires the pool tx to remain in the pool (as soon as it enters the chain, the wallet will identify the tx as a spend correctly).

Expected behavior: the pool tx should be identified as a spend, and any spent outputs should no longer be usable. The wallet state on restore should match the wallet state of the wallet that originally sent the tx.

Identified by @nahuhh in testing the FCMP++/Carrot integration (the bug is unrelated to the integration, I reproduced the issue on current master).

The wallet should already be computing key images during the chain scan on restore, and identifying the spend when reading the pool txs. So `rescan_spent` shouldn't be necessary (nor the additional call to `compute_unknown_key_images_locally`)

IIRC ... the scanner scans the pool first and then scans the chain, which would explain why it's missing spends in the pool (needs key images from receives when scanning the chain first before being able to identify the spend in the pool). The wallet also uses `m_scanned_pool_txs` to cache already scanned pool txs, so perhaps the reason it's missing these spends is because it already cached them as scanned before it had identified the received outputs (that those spend txs spent them in). There is also the whole incremental approach to pool scanning (give me all new pool txs since I last asked) where the daemon may not be returning the same pool txs it already returned.

If above is the case, which should be tested/confirmed before implementing a fix, maybe it makes sense for the wallet to take the extra steps necessary to re-scan the pool if it had just scanned >= 10 blocks (clear local cached variables preventing it, and then just re-executing `refresh`). I say 10 blocks because that's the minimum necessary for pool txs to be able to spend outputs in the chain (EDIT: double check off-by-1).

## nahuhh | 2025-09-13T23:24:02+00:00
This issue can be reproduced in a few different scenarios, and lead me to believe that this issue is caused by the wallet neglecting to parse the pool for its own spent key images.

As stated in OP, this can be triggered by rescanning a wallet while it has spends in the txpool.

this can also be triggered by simply using multiple instances of the same wallet,

To reproduce the latter:

Wallet A and Wallet B have the same seed, and could even have the same (copy) cache file.

1. Open both wallets
2. Send a tx from Wallet A
3. Refresh Wallet B. Wallet B sees the tx (and even requests your password) but does not scan the tx into the wallet. 
4. Observe that Wallet B does not reduce the balance to match Wallet A
5. Try to send the same amount using Wallet B
6. Double spend error

This confirms that wallets are only accounting for incoming txs when peering into the txpool. Outgoing txs are not added to the tx history until a block is mined.
