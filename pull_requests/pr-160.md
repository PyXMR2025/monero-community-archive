# Original Description
Separates the database abstraction layer & the Monero/blockchain-related code into:
- `cuprate-database`
- `cuprate-blockchain`

respectively, with the latter being built on-top of the former.

# Discussion History
## hinto-janai | 2024-06-24T21:15:43+00:00
## `cuprate_database::Env` 1/2

`cuprate_database` no longer has any idea what the user is doing since it is _only_ a database abstraction. This means it doesn't know what tables the user wants, so this invariant is passed on to users now.

This means after `cuprate_database::Env::open`, you have a database but with no tables and it is up to you to create them (where as before `cuprate_database` created all the blockchain tables).

## hinto-janai | 2024-06-24T21:19:25+00:00
## `cuprate_database::Env` 2/2

There is a new function: `create_db`, it just creates the passed `Table`.

The existing:
- `EnvInner::open_db_ro` will error if the table doesn't exist
- `EnvInner::open_db_rw` will create the table if it doesn't exist. This is nice side effect since both `redb/heed` do this but it is documented for clarity that `create_db` should be used just in case

## hinto-janai | 2024-06-24T21:23:49+00:00
## `cuprate_database::Table` 3/4
So, `cuprate_blockchain` defines the type/function that opens all tables.

This new trait `OpenTables` is basically just a copy of the previous:
- `EnvInner::open_tables`
- `EnvInner::open_tables_mut`

It is generically implemented on `EnvInner` such that `self.method()` notation works:
```rust
use cuprate_blockchain::OpenTables;

env_inner.open_tables();
```
