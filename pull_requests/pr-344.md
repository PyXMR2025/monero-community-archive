It could just be put into a tokio task imo.

The docs on `stdin`:

> Constructs a new handle to the standard input of the current process.
>
> This handle is best used for non-interactive uses, such as when a file is piped into the application. For technical reasons, ***`stdin` is implemented by using an ordinary blocking read on a separate thread***, and it is impossible to cancel that read. This can make shutdown of the runtime hang until the user presses enter.
>
> ***For interactive uses, it is recommended to spawn a thread dedicated to user input and use blocking IO directly in that thread***.

I highlighted the important parts, TL/DR tokio uses a seperate thread for this as well, and recommends people spawn their own thread for interactive uses.

## hinto-janai | 2025-01-09T15:41:29+00:00
This applies to any `'static` data but these channels can be leaked such that edge-cases with thread-exit order + channel send/recvs don't panic. Another method is returning on channel errors in `command_listener` and `io_loop`.

Contrived yet possible panic:

- `command_listener` thread receives command
- CTRL+C is received
- `io_loop` thread exits
- `command_listener` thread panics on `blocking_send()`

Another example (more likely):
- CTRL+C is received
- `command_listener` thread exits before `io_loop` thread
- `io_loop` thread panics on `recv()`
