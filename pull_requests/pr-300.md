## hinto-janai | 2024-09-26T21:27:03+00:00
I thought we would be replacing embedded JSON within RPC responses with these types but I realized: it's not actual JSON, it's a string promising it contains JSON.

I.e. the RPC response is:
```json
{
  "json": "{\n  \"version\": 1, \n  \"unlock_time\": 0}"
}
```
not:
```json
{
  "json": { "version": 1, "unlock_time": 0 }
}
```

I don't think there's any reason it shouldn't be actual JSON other than some internal `monerod` detail making that hard to do. Anyway, we have to keep this a `String` but we can still use the JSON types to be more type-safe.

A single block containing 200 transactions puts `~1.152` kilobytes of hexadecimal data on the stack.

I don't think we'll run into stack overflows unless we're somehow holding many many blocks (500-1000) in a stack frame at once? I think it'll be safe to use `HexBytes` within RPC types as well.
