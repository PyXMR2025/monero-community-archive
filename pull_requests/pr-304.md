What are these `{{}}` values? Shouldn't it be:
```toml
peer_save_period = { secs = 90, nanos = 0 }
```
If it's because we're deserializing user input _directly_ to our `Config`'s `Duration` then this is where I've used different structs in the past. `ConfigBuilder` gets read from disk + merged with args, then that gets mapped to an internal `Config`.

1. Handle/parse CLI, create a `ConfigBuilder` [^1]
2. Read `ConfigBuilder` from disk (or `Default::default()`) [^2]
3. Merge CLI + disk `ConfigBuilder` (CLI values > disk values) [^3]
4. Map and sanity check `ConfigBuilder` into a valid `Config` if possible or exit [^4]
5. Set `Config` globally as read-only [^5]

We don't have to do it like this exactly but the `ConfigBuilder` -> `Config` mapping makes the hacky parsing unnecessary. 

The sanity checks are for a few things:
- some options can be in opposition with each other
- invalid values that only appear post-merge
- some values must be post-processed (e.g. reading from a PATH)
- some options require other options to also exist (e.g. TLS cert + key)

In general the builder fields are user-friendly and `Config` is specific for internal needs, e.g. `peer_save_period` would be something like `u64` treated as milliseconds - this gets mapped to `Duration` in the `fn(ConfigBuilder) -> Config` function.

This can be a TODO but it's a tradeoff:
- If left as TODO, someone will have to rewrite all of this code before any public release
- If not, you'll have to make lots of changes here and this PR will stall
