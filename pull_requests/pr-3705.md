# Original Description
The interactive daemon commands `limit`, `limit_up` and `limit_down` plus the corresponding RPC calls were basically broken since my own [PR #2866] was merged last December: When I corrected a bug and streamlined code in connection with the command-line parameter `--limit-rate`  I did not recognize that the code for the interactive commands and the RPC calls behind them needs also corrections:

Any limit given interactively or over RPC was still multiplied by 1024 which is wrong since [PR #2866].

Those multiplications confusingly look like they could have something to do with going from kilobytes/second to bytes/second and therefore needed, but in fact they just were part of some "fudging" necessary because of a then still unrecognized bug. Details see my December PR.
