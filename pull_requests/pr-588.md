# Original Description
### What
Add sync progress percentage and blocks remaining to the handler's block log messages during sync

```bash
2026-02-24T00:36:53.542450Z  INFO incoming_block_batch{start_height=5572 len=27}: Successfully added block batch (0.15%, 3611097 left) fast_sync=true
2026-02-24T00:36:53.544165Z  INFO incoming_block_batch{start_height=5599 len=27}: Successfully added block batch (0.16%, 3611070 left) fast_sync=true
2026-02-24T00:36:53.545870Z  INFO incoming_block_batch{start_height=5626 len=27}: Successfully added block batch (0.16%, 3611043 left) fast_sync=true
```

This PR skips out on adding a time estimate, as the easy solutions will not be accurate due to the difference between pre-RCT and post-RCT block processing time

Addresses #474 

### Why
More info so the user knows how much work has been done and how much is left to process

### Where
- `cuprated`
    - `blockchain/syncer.rs` - adds a format_sync_progress function, receives a atomic usize for shared target height. add `network_tip` function as part of DRYing after the changes
    - `blockchain/manager.rs` - wires the atomic through to the handler
    - `blockchain/manager/handler.rs` - added progress to the log sites

### How
The syncer writes the network target height to a atomic `usize` (set on download start, refreshed every 10s, cleared after all batches drain). The handler calls `format_sync_progress()` at each log site, which returns ` (XX.XX%, N left)` or an empty string
