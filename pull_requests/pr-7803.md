# Original Description
The internal epee DOM is changed from a `std::string` to a `epee::byte_slice` - making some binary copies a ref count change instead. This should help with cache-thrashing a lot (especially on small cache CPUs), since it removes many copies in the block-sync paths. However, the changes amount to a small fraction of the overall sync-time on a ryzen 3900x CPU - so take that into consideration when comparing against how many lines this touched.

* (20 blocks) Binary -> C++ data structures has ~71% time reduction on ryzen 3900x (970 microseconds to 280 microseconds). Expected to be more pronounced on CPUs with smaller caches.
* (20 blocks) C++ data structures copy (occurs in one place in code currently) 65% reduction on ryzen 3900x (5.25 milliseconds to 1.83 milliseconds).

Both timings were averaged over 1000 runs (so everything was in large CPU cache, making the changes appear _less_ significant).

I'll do a quick review of the code to highlight where the performance changes occur.
