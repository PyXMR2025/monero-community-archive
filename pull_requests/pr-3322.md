## stoffu | 2018-03-03T05:16:02+00:00
OK, after carefully reading the code, I still think there's an issue. AIUI, the function `get_outs` works as follows:

- `req.outputs` is filled by `selected_transfers.size() * requested_outputs_count` entries where the number `requested_outputs_count` is 150% of the ring size plus some constant (50).

- After the `/get_outs.bin` query to the daemon, the wallet repeats randomly picking outputs from `daemon_resp.outs` in the range `[i*requested_outputs_count, (i+1)*requested_outputs_count)` for the `i`-th input until `outs[i].size()` equals `fake_outputs_count+1`.

The problem lies in the latter: when the key image has a known ring, `req.outputs` gets initialized with those known ring members. Then, in the subsequent `while` loop from L5845, `req.outputs` is further populated using the random selection algorithm based on triangular distribution, until the number of requested outputs for each input reaches `requested_outputs_count`. Finally, the ring members to be used in the resulting tx is selected randomly from the daemon response and gets written to `outs[i]`. Because of this last random selection step, the resulting ring will likely fail to reproduce the known ring.

I confirmed this behavior by tweaking the number `requested_outputs_count`: when it's set to `fake_outputs_count+1`, the known ring gets successfully reproduced. As `requested_outputs_count` grows further away from `fake_outputs_count+1`, the probability of being able to reproduce the known ring declines.
