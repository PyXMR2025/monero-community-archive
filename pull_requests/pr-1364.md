# Original Description
The `thread_group` class that was added recently can re-use threads, but only "wait" on a single group of tasks at a time (all outstanding). This could be problematic if it were put into some "core" code somewhere; an asynchronous task could not start more tasks + wait because it would result in deadlock (the spawned task would be waiting for itself).

Eventually I settled on a fork-join implementation. The syntax is stolen heavily from `boost::thread::experimental` and `HPX` code, both of which are based on C++ proposal N4105. A semi-complex example from tests:

```c++
TEST(ThreadGroup, Nested) {
  struct fib {
    unsigned operator()(tools::thread_group& group, unsigned value) const {
      if (value == 0 || value == 1) {
        return value;
      }
      unsigned left = 0;
      unsigned right = 0;
      tools::task_region(group, [&, value] (tools::task_region_handle& region) {
        region.run([&, value] { left = fib{}(group, value - 1); });
        region.run([&, value] { right = fib{}(group, value - 2); } );
      });
      return left + right;
    }

    unsigned operator()(tools::thread_group&& group, unsigned value) const {
      return (*this)(group, value);
    }
  };
  EXPECT_EQ(610, fib{}(tools::thread_group(0), 15));
  EXPECT_EQ(610, fib{}(tools::thread_group(1), 15));
}
```
Each task_region can "fork" an asynchrous task via `task_region_handle::run` any number of times, and control from the region is not returned until all tasks have been completed. If an exception is thrown inside of the region itself, it aborts unexecuted tasks, and waits for all to complete or abort. Additionally, new "regions" can be created inside of spawned task using the same `thread_group` without causing deadlock. The only "catch" is that the `task_region_handle` object cannot be given to another thread, but there is never any valid reason to do so.

Thoughts? See `ringCt.cpp` for how this looks in "real" code.
