## vtnerd | 2020-11-09T21:01:25+00:00
Drop the `&`, this was not a mistake or oversight. If this was done elsewhere in the patch too, which is unrelated to the PR title, please revert them. There are numerous internet discussions about using `string_view`/`string_ref`/`span` objects by value instead of reference. The type has 1 pointer and 1 `size_t` value (small), and is trivial to copy and destruct.

A quick summary of the discussions - this type by value requires setting two registers on all modern ABIs, and even on legacy x86 passing the value is fairly simple. In the majority of cases, this object is copying values from a `std::string` or `std::vector` so a `&` forces the compiler to push space onto the stack, copy a pointer+length, then set a register to that stack location. Then the backend optimizer is forced to consider aliasing with other `span` objects of this type (which should rarely be an issue, but nice to avoid where possible). In contrast, by value requires setting two registers and the backend knows the `span<>` type itself cannot be aliased.
