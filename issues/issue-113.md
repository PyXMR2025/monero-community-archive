## MrCyjaneK | 2025-02-13T06:02:09+00:00
Have one step fetch all sources, and then rest of the build should be preformed offline. This is a standard practice in many projects, it will ensure that nothing tries to modify the source/outcome of the build.

One way to tackle this would be to 

1. create a dockerfile with build environment
2. docker run... to fetch the sources
3. docker run --network=none... for the rest of the build (including validation of said packages)

contrib/depends system already allows caching downloads by just executing make download for the most part

```bash
make fetch # Fetch all sources (shallow, pinned commits)

make libcxx     # Cross-compile libc++ (~10 min)
make assemble   # Assemble final NDK layout (~1 min)
```

## Justxd22 | 2026-03-07T03:09:03+00:00
i was looking at building `sysroot` from source, it's divided into 2 parts:

- `.so` stubs (libc.so, libm.so, liblog.so etc.): These are shared libraries. They contain NO real code just a list of function names (symbol table). These are generated by a tool called `ndkstubgen` that reads bionic's `.map.txt` files and creates empty C functions, then compiles them.
- `.a` static archives (libc.a, libm.a, libz.a): These are real compiled code from bionic. They contain actual implementations (518 object files in libc.a alone). When you statically link, this code gets embedded in your binary.

Building bionic (the .a files) requires AOSP's Soong build system, Google's custom build tool that replaces Make/CMake. Bionic is deeply integrated into AOSP. We can't just `cmake bionic && make` You'd need to set up the entire AOSP build environment, which is massive but doable.

Still unclear weather monero actually needs them `.a`
