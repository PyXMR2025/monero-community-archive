You should really regenerate the makefile with cmake:

```

mkdir build
cd build
cmake -DNO_AES=ON ..
make
./bin/bitmonerod
```

## radfish | 2016-06-01T22:00:49+00:00
Thanks for details. It's absolutely that I/O is the bottleneck. I've been fighting with several issues that manifest as extremely excessive I/O to the point where the node stops working. The good news is that when the node correctly operates in steady state, it uses negligibly little I/O.

For fast syncs, you can sync on an armv7 machine (and in the future any machine) with faster storage and copy lmdb folder. I successfully used ARM server at scaleway.com for this purpose.

Yeah, to fix the current issue, cmake script needs to be updated to set the NO_AES for ARM. (I don't think per-architecture targets in the top-level makefile is a good approach, but the current target for ARM would need to be split into ARM 32 and ARM 64). IMHO, the top-level makefile should be removed altogether; all flags that need to be set on specific architectures need to be determined by CMakeLists.txt script.
