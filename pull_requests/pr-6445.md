## vtnerd | 2020-04-14T21:15:22+00:00
The transfer test is now verifying that the wallet initiated send is "hidden" from restricted RPC calls. The wallets are hard-coded to a specific daemon at startup - the `Wallet()` constructor does not launch a new wallet process. So the wallets are initiating a send, and then calls are made to the unrestricted and restricted RPC ports to verify the different states.

Alternatively, the new daemon (index 2) could be launched that connects via p2p to the original daemon (index 0). There are two difficulties with this mode. (1) The daemon-0 must connect to daemon-2 (for outgoing Dandelion++ stem), and daemon-2 p2p must be listening when daemon-0 launches (daemon-0 tries every second but the test is a bit more racy unless a wait for daemon-2 is added). This can be done by launching in reverse order with some sleeps and port probing. The (2) second issue that if daemon-0 rolls "fluff", then the forwarding to the next daemon is randomly delayed by 1-4 seconds 95% of the time (there are no _actual_ bounds since its purely exponential). Additional sleeps in the transfer test could be added to handle this and be mostly effective, but they would need to be more than 4 second sleeps.

Adding new wallets seemed easier. Doing the alternative would (kind-of) functionally test Dandelion++. Realistically would need more daemons to fully test it though.
