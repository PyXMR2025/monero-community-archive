## hinto-janai | 2024-06-10T20:32:55+00:00
It's also worth noting we error instead of ignoring extra fields, unlike `monerod` and other [JSON-RPC 2.0 impls](https://docs.rs/jsonrpsee-types/0.23.1/src/jsonrpsee_types/response.rs.html#198).

I.e.:
```json
{"hello":0,"jsonrpc":"2.0","id":"0","method":"get_block"}
```
This is OK for `monerod` but will error for us. I think we need to do a manual deserialization impl to support this.

EDIT: [nvm serde ignores extra fields by default](https://serde.rs/container-attrs.html#deny_unknown_fields). This means to be compliant with no `result` and `error` we must use `#[serde(deny_unknown_fields)]`, which means we will error on extra fields...

| Impl        | Allows any case for key fields excluding `method/params` | Allows unknown fields in main `{}`, and response/request objects |
|-------------|--------------------------------|-------------------------------------------------------------|
| `monerod`   | ✅             | ✅
| `jsonrpsee` | ❌             | ✅
| `json-rpc` with manual serde impl | (we decide) | (we decide)
| `json-rpc` with macro serde impl | ❌ | ❌

## hinto-janai | 2024-06-10T22:31:34+00:00
> we can just rely on people following the spec, and if they don't that is their own problem.

I think we should enforce the spec since this is an invariant that checks future code/behavior as well. On the client side, receiving a `Response` that has both `result/error` doesn't make logical sense, assuming and serializing into one of the fields is incorrect behavior.

I.e. this should error:
```rust
// from some server
let j = json!({
    "jsonrpc": "2.0",
    "id": 0,
    "error": e,
    "result": r
});

// client
serde_json::from_value::<Response<String>>(j).unwrap();
```

> I don't really want to do a custom impl because of the complexity and I think it' necessary that we allow unknown fields

I don't think it's _too_ complex. How does a manual impl for only `Response` sound? We enforce the spec, and also allow unknown fields. `Request` can be done with macros since there's nothing to enforce. We would only allow lower-case.
