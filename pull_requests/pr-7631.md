# Original Description
Under OpenBSD/amd64 6.8, with its boost package, the `-march=native` option causes a compile failure in any file that includes `boost/thread.hpp`. The same is true for `-march=<any other x86-64 platform>`.

Under OpenBSD/arm64 6.8, the `-march=native` flag isn't even supported. This fact is detected, and the flag is already not used by default, so this change has no effect there.

The compilation error can be reproduced as follows:

```
$ echo '#include <boost/thread.hpp>' | c++ -I/usr/local/include -x c++ - -march=westmere
```

It's how cmake works, variables are scoped to the enclosing function or, if not in a function, the current directory. IOW `function` and `add_subdirectory` are the only way to create a new scope. https://cmake.org/cmake/help/latest/command/set.html

## selsta | 2021-03-25T15:40:49+00:00
Why is the first set local to the function and the second is not? cmake always confuses me.

## ndorf | 2021-03-25T15:49:19+00:00
The second one uses the `CACHE` option to the `set` command and so sets a cache variable, which is a totally different thing altogether. (Yes, cmake is, um, special.) These are meant to be user-settable (e.g. with the `cmake -Dx=y` option), persist across runs, and have project-wide scope. See also https://cmake.org/cmake/help/latest/manual/cmake-language.7.html#cmake-language-variables
