# Original Description
Unfortunately, divisors were NACK'ed this last week by Cypherstack. So in all likelihood, the performance, especially verification time, of FCMPs will change significantly. However, we do not know in advance what that might be. So in order to simply move forward with development, #26 has been merged, codifying the API goal that FCMP++ tx weight should a function of solely: A) number of inputs, B) number of outputs, and C) length of tx extra. But the actually value shouldn't be considered final and *should* be revisited, even if we go ahead with divisors.

Here's some analysis of weight properties by @j-berman : https://github.com/seraphis-migration/monero/pull/26#discussion_r2057203539

# Discussion History
## ArticMine | 2025-05-16T02:39:47+00:00
The analysis of the weight properties should take into account how the transaction will be eventually priced with respect to the quadratic penalty using discrete fees. I recognize that this has not been the case in the past where we have allowed all transactions regardless of weight at the lowest fee level. This leads to lighter transactions effectively carrying the burden of scaling by themselves. To understand the issue a transaction 4x the weight attracts 16x the penalty if it is added at the start of the the penalty.  if it only pays 4x the fee it will not be added by the miner, and has to wait until smaller transactions scale the block weight 

My current proposal for scaling FCMP++ was to increase the minimum penalty free zone to 1000000 bytes and the reference transaction size to 10000 bytes. This would allow up to 8in 8out transactions  to pay the base fee, without any add on verification time weights  Heavier transactions either because of a greater number of inputs or large tx extra would pay the next fee level at least 4x. 

We now take a look at the proposed table in https://github.com/seraphis-migration/monero/pull/26 For all transactions over 2 in the actual cost in terms of fee paid increase by an additional factor of 4x. So for example the fee ratio of 1in to 8 in is not 5056/34278 but rather 5056/137112.

For the above reason I propose we dispense with verification time weight penalties and instead just apply the 4x fee thresholds above 8in 8out / 10000 bytes, 40000 bytes etc. An alternative though not my first choice would be to apply the 4x fee threshold above 4in  4out / 8000 bytes. 

In order to maintain the integrity of the incentives anticipated  in the proposed table in https://github.com/seraphis-migration/monero/pull/26 we would have to increase the minimum penalty free zone to 4000000 bytes with a 40000 byte reference transaction size. This has significant downsides when we consider most transactions have 1 or 2 inputs, such as certain spam attacks with light transactions and subsidizing tx extra data storage which is not subject to the verification time weight penalties. This is apart from the fact that we would be double charging the high verification time transactions. 

Instead of the additional complexity of verification time weight penalties we can stick close to the actual size for weights following "tx weight should a function of solely: A) number of inputs, B) number of outputs, and C) length of tx extra"  and let proper pricing of the scaling of the block weight take care of the verification time growth with number of inputs issue.

## ArticMine | 2025-08-13T16:17:51+00:00
The fee calculation is based solely on transaction size and does not factor verification time at all. The reason it is quadratic with penalty itself is quadratic. https://github.com/ArticMine/Monero-Documents/blob/master/MoneroScaling2025-07.pdf section 2a. The formula f for the additional.fee to add a transaction in the penalty zone is FT = RBase(2*B*BT + BT^2 ). For B = 0, when the transaction is added at the start of the penalt this reduces to FT = RBase* BT^2

If the transaction does not meet the threshold fee there is no incentive for the miner to add the transaction in the penalty.  If a spammer spams a large number of transactions that do not meet this fee threshold and the nodes relay them, then the nodes are forced to verify the transactions, while the miners does not mine  them. The result is a free spam attack on the nodes since the fees are never paid. 

In the past we have gotten away with not enforcing this since the large transaction were few and far between and we were mostly outside of the penalty zone. FCMP++ changes this because of the high verification cost of large transactions.  There is one possible relaxation for cases where ML > ZM and that is to reduce the weight of the transaction by ZM/ML for the minimum node relay fee levels 16x and higher since they scale linearly. 

As for the number 2, 11, 28 and 64 they were chosen based upon where the number of inputs caused the transaction size to go over the 10000, 20000, 40000, and 80000 byte thresholds for at least some of the outputs. 

The only other option is see is to increase ZM. The issue here is that there is no pricing in the penalty free zone and past history has shown this is where spam attacks have occurred. 

The reality is that the verification time cost is more than addressed by enforcing the incentive to mine for all transaction weights. 128 input transactions can be feasible under FCMP++. They will however be expensive.

## ArticMine | 2025-08-27T16:19:04+00:00
> Update: @kayabaNerve's latest work optimizing prove/verify demonstrated an improved high input verify, though it's still a bit faster to verify 16x 8-in versus 1x 128-in. Source: https://github.com/kayabaNerve/monero-oxide/commits/fcmp%2B%2B-unaudited-verifier

In addition one can use parallel processing to verify the 16x8in transactions. 

This bring me to the more general question of large transactions. By large I will refer to transactions whose weight is greater than the reference transaction weight of TR = 10000 bytes. As per the proposal for FCMP++ scaling I am using a minimum penalty free zone ZM =  1000000 bytes. TR and ZM are increased from the current values of 3000 bytes and 300000 bytes respectively. The bulk of Monero transactions have 1 or 2 inputs and have estimated weights under 10000 bytes. This makes 10000 bytes an appropriate weight for TR.

How is TR determined? 

The principle is very simple. A miner will only add transactions to a block that are profitable. 

**In order for the blocks to scale the fee paid to the mirier for a transaction must be greater than the additional penalty that transaction attracts.** 

The quadratic nature of Monero's penalty means that if one doubles the weight of a transaction the fee required to scale has to increase by a factor of 4

TR is chosen so that the vast majority of transactions comfortably meet the above requirement. A typical metric is the weight of a 2in 2out transaction which is estimated at 7865 bytes. This supports TR = 10000 bytes. The minimum node relay fee is set so that a transaction of weight TR can scale at the start of the penalty. The current minimum node relay fee scales as the inverse of the  long term median 1/ML. This works because ML is lower bound by ZM. 

If a transaction does not meet the above criteria, and is consequently not economic, the transaction will languish in the transaction pool until there is enough space in the block for a miner to mine it economically. In the past large not economic transactions have been few and far between and the block size has for the most part been well below ZM. This has allowed these large non economic transactions to be mined even though they did not pay a high enough fee to allow them to scale. 

It now becomes apparent how non economic transactions can be used to DDOS the nodes, who relay such transactions, especially if there is also a significant verification time associated with these transactions. An attacker can simply flood the network with not economic transactions. Most of these transactions will not be mined, consequently have no cost to the attacker. 

My conclusion: If it is desirable that fees be linear with  the actual verification time, then it will be necessary for transaction weights to grow as the square root of the verification time. Then when the quadratic penalty is applied the resulting fee to overcome the penalty will be linear with the verification time. This of course means that the weights will end up below the actual size.

## jeffro256 | 2025-10-10T22:33:15+00:00
@spirobel With ZK-proofs, there's a difference between *aggregating* proofs and *batching* proofs. Oversimplifying a bit, aggregating is when the *prover* squishes multiple proofs, proving different things, into one big proof proving all those statements. The big proof is almost always smaller then the sum of the individual proofs (Otherwise we wouldn't aggregate them). Batching during verification is when the *verifier* speeds up the verification of multiple proofs by verifying them all at once. Verifying a large batch is almost always faster than verifying individual proofs (otherwise we wouldn't batch them).

As implemented, FCMP++ txs only have one aggregated membership proof which proves membership for all inputs, which saves a lot on bandwidth. The downside is that we can't batch verify inputs individually, the smallest unit to verify at once is the one membership proof for the entire transaction. Note, however, that we *can* do batch verification across multiple transactions, which is what is done during block sync.

The membership proof is by far the most expensive proof to verify in a FCMP++ transaction. However, for an $N$-input transaction, there are also $N$ *Spend-auth/Linkability* (*SA/L*) proofs. These can be batched across inputs *and* across transactions.

### New proposal for tx weight

**Make the tx weight calculation *roughly* byte size, where you can still calculate the weight of a tx in an offline context using n inputs, n outputs, and extra len.**
  - The fee is assumed to be a constant 8 bytes (max `uint64_t`), even though it's stored as a varint and can be smaller (or technically larger) than 8 bytes.
  - The FCMP++ tree is assumed to be a constant 7 layers (justified [here](https://github.com/seraphis-migration/monero/blob/2c9ebbef2831e038a5c988fa31e9eabfac430927/src/cryptonote_basic/cryptonote_format_utils.cpp#L572-L592)).

**Why back to byte size?**

A major reason that we initially wanted to proportionally increase weight for larger FCMP++ txs is because membership proof byte size does not increase as fast as membership proof verification time. However, as n inputs increases, the SAL proof byte size increases faster than the membership proof. Thus, the increasing SAL proof byte size offsets the increasing membership proof verification time.

  - 1-in/2-out: the SAL proof is ~7% of the tx size, membership proof is ~78%.
  - 128-in/2-out: the SAL proof is ~41% of the tx size, membership proof is ~53%.
  
Weight following byte size is very straightforward to justify, and should have the most straightforward path to rough consensus in my view. It's also simplest to implement.
