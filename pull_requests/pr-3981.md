## vtnerd | 2018-07-05T15:19:18+00:00
> The data is sorted: it's a cumulative count, so can never go down.

The caller is required to sort the values, but this is not documented in the header. Adding this information to the header is more brittle than taking the `std::vector` by value, and doing a sort internally to ensure correctness. Any caller can do a `std::move` to elide the copy entirely. Or a `if (!std::is_sorted(begin, end)) throw ....;` to enforce constraints (although arguable doing the sort is preferable).

I suppose one issue, a large `std::vector`, will add lots of ops ... ? If so, the problem is likely with the API.

> Though if we move to reusing the existing table, that makes height height free.

> separate table: 
> * no wasted memory, good caching, slow to get to a single element
> * reusing existing table: wasted memory, bad caching, ~instant lookup of a single element

What about the case where a `struct {height, value}` is stored as a new LMDB table? Is this function expected to _mostly_ retrieve the **entire** range of values? If so, then this implementation is reasonable. However, the `std::vector` being returned and provided as an argument will either continue to grow (somewhat quickly) as the number of blocks increases OR callers will grab a smaller range thus making the linear search inefficient. Since you commented elsewhere that an RPC for retrieving specific values is going to be added (and since the RPC is already allowed to restrict a range), using the extra space for a proper LMDB key makes sense.
