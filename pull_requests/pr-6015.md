# Original Description
This fixes #6012 and seems to work correctly in all cases. The file is regenerated if and only if the version tag changes (or the input file does). In particular:

1) Nothing is regenerated or rebuilt if nothing changes in the source tree
2) After `git commit --allow-empty -m 'whatever'`, the version file is regenerated and built, and the executables quickly relinked, with no unexpected other targets being built. The new executables report the correct new version hash.
3) Modifying the input file `src/version.cpp.in` or deleting the output file `build/.../version.cpp` causes the file to be regenerated with no unexpected errors.

Additionally, since 33e91e1a1e94c2957f59def9fe49f75956a9f699, I don't think the two cases not using GenVersion.cmake could have worked -- they don't set `VERSION_IS_RELEASE`, without which the generated version.cpp fails to compile (empty space after the bool assignment). That should also be fixed here.
