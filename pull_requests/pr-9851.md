# Original Description
This one is a little aggressive - it adds move semantics to `transaction` and `block`. I deleted the move constructor+assignment to both objects and tried compiling everything but tests. This identified every move operation (that is currently secretly a copy), and I manually verified that the moves were valid. There was one `transaction` case that I could not verify, which I downgraded to a copy.

This PR also changes the default+copy constructors, and copy assignment. The reasoning was the `atomic` values - the changes have fewer memory barriers and branches, which should result in slightly faster code (particularly on arm).

The primary benefit is a `transaction` copy becomes a move in `add_block`. This should have a (small but) measurable improvement in block syncing, etc, as a `transaction` copy is rather expensive (due to the number of allocations).
