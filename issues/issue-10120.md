# Original Description
When calling `get_blocks.bin` with `prune = true` the blocks returned will have their tx blobs pruned but wont set the `prunable_hash`, leaving it as the default. This means you can't reconstruct the tx hash and verify the txs are actually the correct ones provided for the block.

https://github.com/monero-project/monero/blob/b591866fcfed400bc89631686655aa769ec5f2dd/src/rpc/core_rpc_server.cpp#L784

# Discussion History
## kayabaNerve | 2025-09-28T12:24:24+00:00
This will be a blocker for `monero-oxide` using `get_blocks.bin` in it's provided RPC :/ We have unvalidated and validated methods, where the validated methods do ensure the transactions are as expected. This means, in order to build a full SPV client, one would _only_ have to verify the header's proof of work (as we check all data is consistent otherwise).

While ideally, wallets can solely call our RPC function which solely calls `get_blocks.bin`, the inability to check the consistency of the RPC in this location means we'll have to use the JSON-RPC `get_block` (which doesn't work in batch, see #10118) followed by the RPC (JSON but not JSON-RPC) `get_transactions`. This is on the presumption that the 2-4x overhead of JSON-encoded transactions (2x for hex, yet the JSON is not the hex alone, and the other fields/JSON schema itself are non-trivial) is less costly than requesting the full transactions in binary. Having written that assumption out, I believe the JSON-encoded without-prunable transaction may be as large as the EPEE-encoded with-prunable, so `monero-oxide` may simply use `get_blocks.bin` without pruning?

Such an annoyance to work through...

I primarily write this to provide context and explain the real-world impact of these issues.
