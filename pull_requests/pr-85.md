## hinto-janai | 2024-03-08T18:32:59+00:00
This is due to https://docs.rs/redb/latest/redb/trait.ReadableTable.html#tymethod.range.

- `heed` wants `Range: RangeBounds<T::Key>`
- `redb` wants `Range: RangeBounds<impl Borrow<T::Key>>`

The current `redb` code is `&T::Key`. If we switch our key to `Cow<'_, T::Key>` too, our range has to `impl Borrow<Cow<'_, T::Key>>`.

I guess the input would be:
```rust
let range = Cow::<u64>::Owned(1)..Cow::<u64>::Owned(10);
```

This does create a incompatibility when switching backends since `heed` wants a `u64` and `redb` wants a `Cow<'_, u64>`. We could switch it so `heed` also uses `Cow` internally I guess...?

## Boog900 | 2024-03-08T20:37:08+00:00

## hinto-janai | 2024-03-12T01:56:56+00:00
To be more clear:
- `heed` wants `Range: RangeBounds<T::Key>` (`heed` sees `&'a T` as the key)
- `redb` wants `Range: RangeBounds<Cow<'a, T::Key>> + 'a` (`redb` sees `Cow<'_, T>` as the key)

We could switch `heed` so it sees `Cow<'_, T>` instead of `&'a T` as the key type, but means we're wrapping all key/values in it during serialization (not bad, still gets serialized as T). It means our ranges will be like the above though...

Or we can make `redb` accept `RangeBounds<T::Key>` then in `redb`'s specific `get_range()` we somehow transform:
```rust
RangeBounds<T::Key>
// ...into...
RangeBounds<Cow<'?, T::Key>>
```
is this possible?

## hinto-janai | 2024-03-12T20:20:50+00:00
Okay, I solved this by converting `RangeBounds<T::Key>` -> `RangeBounds<Cow<'_, T::Key>>`
in `redb`'s specific impl https://github.com/Cuprate/cuprate/pull/85/commits/b878d9c93b93bb8fad924ec33743a1ec2237ce24#diff-466feb6ae896745bdb76b9de9b6c341bf6baeff9853df5f5026ab72d3bf0610aR59.

A little hacky but it works. It means our API is `RangeBounds<T::Key>` which means:
```rust
// we get to use this...
get_range(0..1);

// ...instead of this
get_range(Cow::Owned(0)..Cow::Owned(1));
```
