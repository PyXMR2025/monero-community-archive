# Original Description
## Summary
- Refactors multisig key exchange (kex) and account generation.
	- The key exchange interface has been generally simplified.
		- No more `finalize_multisig()`.
		- Most kex functionality has been sequestered in the `multisig` component library, so `wallet2` and friends only need a few relatively simple wrappers to get things done.
		- Key exchange messages contain 'round' numbers, indicating which 'round' of message exchange those message correspond to. 'Rounds' are synchronous phases of key exchange where you collect message from other participants. Recording round numbers in messages allows kex participants to more easily track the kex process.
- The multisig kex algorithm has been updated.
	- Use robust key aggregation (a.k.a. MuSig-style; see [MRL-0009](https://web.getmonero.org/resources/research-lab/pubs/MRL-0009.pdf)).
	- Validating intermediate keys during key exchange is much more robust.
	- The multisig 'base' keys from the initial kex msgs are now used as 'participant identifiers'. All messages from one participant will be signed by the same base key, including during transaction building (backward compatibility **should** be maintained with old multisig wallets).
	- Allow 1-of-N multisig wallets (M >= 2 was an arbitrary constraint).
- There are miscellaneous documentation and readability improvements related to the multisig implementation.

## UkoeHB | 2021-12-10T19:43:35+00:00
This function is 'sanitize pubkeys', it is only responsible for filtering data out of the input messages. The 'evaluate msgs' function does all these kinds of data checks.

> (1) ensure the remote party produced a `public_key` identical to the one produced locally

I explicitly don't do this test so 'round boosting' can work seamlessly (future PR). I assume you are referring to intermediate kex keys, not the final aggregate key.

Why isn't it a problem?
- If there are more rounds left, then we don't care if remote parties have the same keys as us. We take the messages from this round, remove all keys that match local keys, then make DH secrets for the next round. In the next round, we _don't care_ about our keys from the previous round.
  - If the message we make is for the last round, then we save the DH secrets as our final secret keys (who cares if other participants have these keys or not? if they lost their keys, then I guess the local signer just has more signing power than expected).
- If we are in the last round (we will make the final aggregate key from the set of messages at hand), then we already have all our private keys corresponding to signer-sets of size `N - M + 1`. All we care about is if _every set of size `N - M + 1` that excludes us shares a key_. If that condition holds, then if any set of `N - M` signers (other than the local signer) are removed, then there is always at least one person remaining with each key piece. That is the basic requirement of multisig: remove `N - M` signers, and the remaining `M` signers can always complete a signature (note: it is nonsensical from a security p.o.v. for the local signer to validate whether removing himself will allow the other signers to complete a signature).

> (2) that all of the messages came from the same signers

This check is performed in `evaluate_multisig_kex_round_msgs()` where I comment `// 2 (continued). only expected signers should be recommending keys`. The `signers` set is assumed to be constant between kex rounds.

> (3) a check to ensure the aggregate pubkey is identical across all parties

If there are fewer than `N - M + 1` malicious signers, then identical aggregate pubkeys between honest signers are implied, because each signer signs their kex messages. If there are `N - M + 1` malicious signers, then validating the aggregate pubkey is futile because your setup is compromised.
