## Boog900 | 2024-03-20T14:05:35+00:00
`install` blocks the thread until the task is complete, so we can't use that.

The docs for [install](https://docs.rs/rayon/latest/rayon/struct.ThreadPool.html#method.install) says:

> Executes op within the threadpool. Any attempts to use join, scope, or parallel iterators will then operate within that threadpool.
Warning: thread-local data

> Because op is executing within the Rayon thread-pool, thread-local data from the current thread will not be accessible.
Warning: execution order

> If the current thread is part of a different thread pool, it will try to keep busy while the op completes in its target pool, similar to calling [ThreadPool::yield_now()](https://docs.rs/rayon/latest/rayon/struct.ThreadPool.html#method.yield_now) in a loop. Therefore, it may potentially schedule other tasks to run on the current thread in the meantime. For example.

So I think what the `install` method is for is if you are currently in a rayon thread and you want to put work onto another thread pool and not do the work in your threadpool. `install` does not block rayon threads instead it repetitively calls `yield`. You would not be able to use the `spawn` method in this scenario as spawn does not wait for the task to complete. 

And the docs for [ThreadPool](https://docs.rs/rayon/latest/rayon/struct.ThreadPool.html#) says:

> Use a [ThreadPoolBuilder](https://docs.rs/rayon/latest/rayon/struct.ThreadPoolBuilder.html) to specify the number and/or names of threads in the pool. After calling [ThreadPoolBuilder::build()](https://docs.rs/rayon/latest/rayon/struct.ThreadPoolBuilder.html#method.build), you can then execute functions explicitly within this [ThreadPool](https://docs.rs/rayon/latest/rayon/struct.ThreadPool.html) using [ThreadPool::install()](https://docs.rs/rayon/latest/rayon/struct.ThreadPool.html#method.install). By contrast, top level rayon functions (like join()) will execute implicitly within the ***current thread-pool.***

So I think calling `spawn` instead of `install` is fine here.

I made a little test to make sure, if the par_iter in `task` uses the global pool then the thread names will be `None` as we don't set them.

```rust
use rayon::prelude::*; // 1.9.0

fn main() {
    let pool = rayon::ThreadPoolBuilder::new().thread_name(|i| format!("R.{i}") ).build().unwrap();
    
    pool.spawn(|| task("local"));
    println!("spawnd local pool");
    
    rayon::spawn(|| task("global"));
    println!("spawnd global pool");

    std::thread::sleep(std::time::Duration::from_secs(2));
}

fn task(pool: &str) {
    let a = [0; 10];
    
    a.par_iter().for_each(|_| println!("{pool}, {:?}", std::thread::current().name()))
    
}
```
playground link: https://play.rust-lang.org/?version=stable&mode=debug&edition=2021&gist=b37025375f480bd2525da56e3389eb23
