# Original Description
https://github.com/monero-project/monero/blob/5fbfa8a65663e807c6500ae9485e898df9b7c470/src/cryptonote_core/blockchain.cpp#L4396

`blocks.resize(blocks_entry.size())` is used here, but the function could early terminate before the 2nd loop if `block_exists` is true leaving un-parsed blocks in the `blocks` vector.

Since the vector is resized, and if you have partially parsed blocks- you'd not trigger this line

https://github.com/monero-project/monero/blob/5fbfa8a65663e807c6500ae9485e898df9b7c470/src/cryptonote_protocol/cryptonote_protocol_handler.inl#L1221

Where further down that function, the code seems to iterate over the vector assuming all the blocks inside are parsed validly.

All other instances of prepare_handle_incoming_blocks seem to be used purely for 1 block (like mining a block and pre-processing it, or receiving a block when you are synced to the tip- so 1 fluffy block at a time etc.) which wouldn't be affected by this.

The function in question is a bit confusing to read, so I'm not too sure this doesn't break anything. The 2 loops seem to suggest that thread pooling was the intention here, in which case the PR needs to be reworked.
