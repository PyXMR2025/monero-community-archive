# Original Description
The first commit in the list below contains a fix to an issue I introduced earlier when fixing the native build on ARM. Another commit is a revert that brings back the release-static-arm\* targets: it was wrong to delete them -- I misunderstood their purpose: they are not for bulding native binaries (-march=native) on ARM, which I fixed earlier to build  via normal 'make release', but for building generic binaries portable across the given ARM family for release to users. Could call it "light cross-compilation", where there is a constraint that host CPU arch must be compatible with target CPU arch. Same with the release-static-{32,64} targets and *-win{32,64} targets -- on Windows there is no "native" build at all, the only build for Windows is a cross-compilation.

For native builds, ARCH is not set (and defaults to 'native', so '-march=native'). For the "cross-comilation" builds, ARCH is set to the target architecture. There is also an option to set ARCH to "default" to not pass -march at all, but that is currently not used.

In the future, if/when cross-compilation from x86_64 to armv\* is setup, we'll need another set of targets. At that point would either revise the naming convention or add logic to top-level makefile that would detect the host arch and do the right thing to build the specified target arch binary. For example, so that same make release-static-arm7 works if invoked either on an arm box or an x86_64 box.

Editted readme to hopefully make things clearer, shorter, and easier to follow. Link to the editted version without diff noise: https://github.com/radfish/bitmonero/tree/PR--cmake-arch-handling
