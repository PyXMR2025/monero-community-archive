## moneromooo-monero | 2018-01-22T23:03:55+00:00
On second thought, this should use the "worst case" fee, since the actual tx may end up there.
The worst case is if your tx is an integer number of kB minus 1 byte, in which case it'll pay a comparatively low fee per byte compared to the average, which is what the above is checking.
So we need to check something like (fee_per_kB * 12 / 13) /1024, roughly.


## stoffu | 2018-01-23T01:19:17+00:00
I'm not necessarily in full grasp of your point, but updated the code as suggested anyway. Does the number 12/13 come from the typical size of 2-in/2-out txes?

## moneromooo-monero | 2018-01-23T13:47:59+00:00
Yes, typical tx is 13 kB.
With a x monero/kB fee, a 13 kB tx pays the same fee as a 14kB-1 byte one, but has a higher fee/byte ratio. So if you send the 14 kB-1 tx, it always gets pushed back to the end of the queue. The previous code was using the level for a 13 kB tx, so would not see the backlog caused by the 13 kB + some bytes ones, which the actual tx would likely be one of.
