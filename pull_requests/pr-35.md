# Original Description
## Part 1
Next -> https://github.com/Cuprate/cuprate/pull/60

This is the 1st in a chain of PRs that will re-implement `database/`.

Each PR will focus on a specific scope for review purposes.

This one is for:
- general design decisions
- broad file structure
- data type definitions
- small stuff like lints, naming, etc

## Goals
The main goal for this re-implementation is allow for a clean separation between the specific operations other parts of Cuprate would like to perform (`get_block()`, `add_block()`) and the underlying database (`LMDB`, `sanakirja`, etc) such that the callers do not know - and do not care about the actual underlying database.

A benefit this will allow is for Cuprate to "hot-swap" databases, ideally simply by building with a different `--feature`:
```bash
# Build Cuprate using heed.
cargo build --release --feature heed
```
```bash
# Build Cuprate using sanakirja.
cargo build --release --feature sanakirja
```

After testing to see which database has the best characteristics for the types of operations Cuprate performs (considering latency, throughput, parallelism, etc), a "one true" database will be selected as the default, although the underlying harness to quickly swap to another one or even add more in the future will always be there.

This database "layer" must also:
- Be interfaced as a `tower::Service`
- Not block the async runtime

## Divergence from `monerod`
Cuprate's database will diverge from [`monerod`](https://github.com/monero-project/monero)'s in the following ways:

1. Schema will most likely be different
2. Compatibility with `.bitmonero` as-is is low priority 

## Database to implement
The database implementations in mind (for now) include:
- LMDB (via [heed](https://github.com/meilisearch/heed))
- [sanakirja](https://docs.rs/sanakirja)

Both of these will be forked and maintained by Cuprate as there are `monerod`-specific things that must be added (e.g output lookup using [sub-keys](https://github.com/monero-project/monero/blob/ac02af92867590ca80b2779a7bbeafa99ff94dcb/src/blockchain_db/lmdb/db_lmdb.cpp#L3447C1-L3449C80)).

## Layers
This system is logically abstracted into 5 layers internally.

Starting from the lowest layer:

| Layer | Name | Description |
|-------|------|-------------|
| 1     | Database | This is the actual database, or a thin Rust shim on-top of the database that calls database operations directly, e.g `get()`, `commit()`, `delete()`, etc
| 2     | Trait | This is the `trait` that abstracts over _all_ databases and allows keeping the function signatures and behavior the same but allows for swapping out databases; each database will have this implemented located in `src/backend/`, with each database (`LMDB`, `sled`, etc) having its own file defining the mappings. This `trait` is meant to cover all features across databases, and will have provided methods that may not necessarily be the most efficient - if a database can implement a method in a better way, it is re-implemented and will shadow the provided version
| 3 | Abstract Database | This is a concrete object and handle to _some_ database that implements the generic `trait`
| 4 | Thread | This is the dedicated _writer_ thread that is the logical owner of the abstract database. It acts as a sort of "write queue" that maps channels messages from other Cuprate crates into database _write_ calls into eventual responses to the callers via a channel
| 5 | Services | This is the public API that other Cuprate crates will interface with; the abstract database will have `tower::Service<ReadRequest>` implemented where `ReadRequest` is an `enum` with all the specific high-level read-only requests other Cuprate crates need e.g. `get_block()` and `tower::Service<WriteRequest>` where it is the same, but with writes e.g. `add_block()` - the callers will be responsible for execution if this is a read (or perhaps a request is sent to a "reader" thread-pool?), and write operations will be "queued" by sending them to "the thread" which eventually returns the result of the function

## The `tower::Service`
The `tower::Service` will act as the public API for the database that other parts of Cuprate will call. The service will contain an async channel that requests can be sent down.

They only care about making a database request and receiving a result - they have no view into the internal implementation.

A request will look like:
```rust
struct Request {
  request: DatabaseRequest,
  response_chan: tokio::sync::oneshot::Sender<Result<DatabaseResponse, Err>>
}
```

## Parallelism
The databases planned to be implemented all support MVCC, so Cuprate itself doesn't necessarily need to bake-in any synchronization.

Readers scale per core (in ideal situations) with MVCC-style databases, and as such, Cuprate will spawn, control, and maintain a pool of threads (outside of `rayon`, most likely manually spawned) that will handle multiple read and write requests.

These database threads will operate in a `work-stealing, lockless-queue` fashion, where readers/writers outside of `database/` send some requests, and the database threads:
1. Pop the request out
2. Do the work
3. Send result to the requester

## Messages
If we were to go with a split read/write service we would need 4 `enum`'s:
- A read request enum which is given to the read services
- A write request enum which is given to the write service
- A joint read/write enum which the services send to the channel
- A response enum which is returned down the oneshot

## Previous Ideas
These are previous design ideas for the database implementation that were ruled out.

```
I = Idea
R = Reason for being a bad idea
```

**I.** All outer Cuprate crates send/receive *all* database-related requests (via channels) to a single "database" thread that logically owns the database object
**R.** This effectively limits concurrent readers to 1, as all requests/responses would be funneled through a single thread

**I.** Just put the DB operations in the async runtime
**R.** By doing this we risk taking out an async thread from the pool if the DB blocks
