# Original Description
Probably this isn't a comprehensive solution to the "bootstrapping node advertising" idea proposed in #2204, but I just wanted to try something simple anyway...

This patch adds a flag `--bootstrap-daemon-address` to `monerod`, for example:

    monerod --bootstrap-daemon-address node.moneroworld.com:18089

If this parameter is specified, your local daemon connects to that remote node and simply acts as a 'pipe' while it is synchronizing with the network. When the sync is complete, it stops this piping and starts to respond to RPC requests using its own blockchain. The wallet connects to your local daemon as usual, and doesn't know that it's using the bootstrap remote daemon. This can be conceptually seen as moving the remote daemon address parameter `--daemon-address` of the wallet CLI/RPC/GUI to the parameter of your local daemon.

I've also tried another implementation approach (https://github.com/stoffu/monero/commits/daemon-bootstrap-old2) where such a flag is added to the wallet's side, and the wallet would create another instance of `epee::net_utils::http::http_simple_client` and switch between the two daemons as appropriate, but I gave up this approach since it seems to add more complication to the code.

## leonklingele | 2018-01-28T15:46:32+00:00
The problem I'm seeing here is that currently you acquire an exclusive lock on `m_bootstrap_daemon_mutex` for the whole call to e.g. `core_rpc_server::on_getblockhash`. This degrades performance quite a bit as you keep the lock for longer than required. This could be solved by either using two different mutexes (one for `m_was_bootstrap_ever_used`, another for `m_should_use_bootstrap_daemon`) although this might end up being a bit too complicated.

It already helps if you just try to keep the mutex for as short as possible:

```cpp
m_bootstrap_daemon_mutex.lock();
res.was_bootstrap_ever_used = m_was_bootstrap_ever_used;
m_bootstrap_daemon_mutex.unlock();
```

and

```cpp
bool use_bootstrap_daemon;
m_bootstrap_daemon_mutex.lock();
use_bootstrap_daemon = m_should_use_bootstrap_daemon;
m_bootstrap_daemon_mutex.unlock();

if (use_bootstrap_daemon)
{
      res = "This command is unsupported for bootstrap daemon";
      return false;
}
```

Even better than using a `boost::mutex` (which only supports `locking`) you could use a `boost::shared_mutex` which supports read & write locking (a semaphore). That is, when being read-locked (`shared_lock`), another thread trying to read-lock it doesn't have to wait for it to be unlocked again (as multiple readers are allowed). However, when being write-locked (`unique_lock`), no other reader / writer can acquire the lock. This improves performance as it allows for multiple concurrent readers.

__Note__: I haven't done any benchmarks of the current implementation, so I don't know how much of a performance issue this currently is.
