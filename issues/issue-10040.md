# Original Description
To stop selfish mining that has <50%, softly-require timestamps to be accurate. This works because an attacker has to assign a timestamp and solve the block before he knows when he'll need to release it. From a consensus theory point of view, it works because it increases node synchronization which is one of 3 requirement that are necessary to get consensus with only 51%.

It will block selfish mining as soon as the hashrate using this rule is at least as much as the attacker's hashrate. In short, this simply enforces PoW better by softly-requiring timestamps to be more accurate when a node first sees than the current future time limit and MTP. It doesn't permanently rejects bad blocks based on bad timestamps because PoW needs to make the final decision in case there is a real network partition that lasts for more then 2 block times.

Rule to stop selfish mining as soon as miners agree to adopt it:

**Do not accept a block as valid for 2 block times if it timestamp is more than +10 seconds from local time into the future, or more than -15 seconds into the past at the moment the node first sees the block. The extra 5 seconds on the past time limit is to allow for propagation delays. Continue to reject a block after the timeout period if it's timestamp is still >10 s in the future. Honest node clocks must have less than +/- 5 seconds error to prevent their blocks from being rejected and to prevent them from working on the wrong chain. A mining node who's been disconnected and rejoining the network doesn't enforce the past time limit on new blocks until he's synced back up with the longest chain he can find.**

The constants are chosen based on estimates and theory, not arbitrarily by me. They assume nodes can keep time only to within 5 seconds and that typical, maximum propagations delays are up to 5 seconds. These constants can be relaxed by a specific node if it can't achieve them. It only stops working if the attacker plus 1/2 of the nodes ignoring the rule (aka "agnostics") exceeds 50%, That would also make it disadvantageous for a node to follow it. Notice that no data on the block chain is affected. I chose 2 block times for the timeout period, but from the math, it only needs to be ln(2) * blocktime to make sure a selfish miner will lose at least half the time.

A node just rejoining the network would not enforce this rule until he's synced back up with the longest chain he can find. If this causes him to join the attacker's chain, it's not a problem because this rule means he'll switch to the honest as soon as the honest chain has more work because all new blocks in the honest chain after he joined will have good timestamps.  

If a node invalidates a block based on the timestamp, then turns off for a minute before rejoining, the block is still invalid if the timeout period hasn't passed when he rejoins, so he does not necessarily join the longest chain. But if he's off for a period longer than the timeout, he chooses the longest chain, if the tip's timestamp is not more than 10 seconds into the future. 

For 2.5 minute blocks, relaxing the 10 seconds to 20 seconds should prevent most problems. Making the timeout period longer has the disadvantage of causing "honest" partitions in the network to take longer to determine the PoW winner. The timeout period can be as low as the median block time ln(2) * 2.5 minute = 1.7 minutes and a <50% attacker can't expect to make a profit over selfish miners. It's still possible to make a small profit even with this rule for protection if a large attacker assigns timestamps 10 seconds into the future which allows him to hold a block back for up to 20 seconds.

## zawy12 | 2025-08-19T01:59:18+00:00
Oh right, doh, "post date" is correct as in checks.

No node should ask any other node, or check the blockchain, or agree with other nodes on what their mutual source of time is.  As Lamport's original BFT paper said, BFT consensus can't be achieved if the clocks aren't synchronized within some bound and "this is a problem as hard as BFT itself".  In other words, PoW absolutely depends on it, and there's "no solution" and "every solution" in the sense that you can do it any way you like, but you better not ask any centralized source for time. Any coin using median of peer time is vulnerable to a Sybil attack which is why bitcoin removed it. A node operator must ask his cell phone, use a linux or windows NTP, his GPS, his TV, the NIST, or grab a telescope and check the stars...any way he likes...as long as he gets an accurate time and not agree with other nodes about how he might go about doing it.  There's no PoW without a secure clock, and there's no secure clock unless you look at the stars, and in that case, you're still trusting that Earth's orbit and rotation haven't changed.  You also need to know the centralized international committee that defines UTC hasn't redefined it, or that everyone knows if they do. The only 2 data points in this universe that I know of that we can all check today and agree on without trusting anyone is which chain has the highest PoW and what the current local time is.

> The strategy employed in this case, where all 6 blocks were replaced at once, is a variant known as stubborn mining (or a similar aggressive/optimized form of selfish mining). In this approach, the attacker withholds all discovered blocks in a private chain, continuing to mine on that fork even if it temporarily falls behind the public chain in length (i.e., ignoring the usual rule to switch to the longest known chain). The entire private chain is then broadcast only when it surpasses the public chain's length, forcing the network to adopt it and orphan a large number of public blocks in a single reorganization event. This differs from the classic selfish mining strategy, which typically involves more conservative, incremental releases (e.g., publishing one or a few blocks at a time when the lead is challenged) to minimize risk and maximize steady profits, often resulting in smaller, one-by-one orphans rather than deep reorgs. The stubborn variant is riskier—relying on luck to regain and maintain a lead without losing the private chain—but it's effective for causing significant disruption or demonstrating capability, as seen here with Qubic's "more sophisticated" tactic on Monero.

> Stubborn variants like trail-stubborn or lead-stubborn can yield up to 25% higher gains than classic selfish mining, particularly when γ is high (e.g., γ=1) or when combined with eclipse attacks (isolating nodes to control propagation).

> Specific sub-strategies, such as Equal Fork Stubborn Mining (EFSM), are superior in certain parameter regions after difficulty adjustments, while Lead-Stubborn Mining (LSM) excels in narrower domains.
