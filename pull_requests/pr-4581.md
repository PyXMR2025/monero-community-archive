# Original Description
### Problem

`DaemonManager::start()` launches monerod with `QProcess::startDetached(program, args)` (the static overload) and, on Unix, also passes `--detach` to monerod itself. `P2PoolManager::start()` does the same with the instance-level `startDetached()`. In both cases the `QProcess` object is not attached to the real child:

1. The `readyReadStandardOutput`, `readyReadStandardError`, and `stateChanged` signal connections in `DaemonManager::start()` **never fire**. The daemon console in the GUI therefore never receives monerod's stdout/stderr, and `daemonStopped()` is never emitted when monerod exits.
2. Shutdown in `DaemonManager::stopWatcher()` and `P2PoolManager::exit()` falls back to `pkill` / `taskkill /F /IM`. Those commands kill **every** monerod or p2pool on the host, including instances the user runs outside the GUI (e.g. a systemd-managed monerod, or a separately launched p2pool).

### Fix

- Use `setProgram` / `setArguments` + `start()` + `waitForStarted()` on the owned `QProcess` in both managers.
- Drop `--detach` from the monerod arguments on Unix. `--non-interactive` is already passed unconditionally and is what keeps monerod from reading stdin. Running monerod in the foreground under `QProcess` matches the `Type=simple` systemd unit recommended upstream.
- Move the `stateChanged` `connect()` above `start()` so no early state transitions are missed.
- Replace the shutdown fallbacks with targeted calls on the owned process:
  - Daemon: `m_daemon->kill()` (under `m_daemonMutex`).
  - P2Pool: `terminate()`, `waitForFinished(5000)`, then `kill()` as a safety net.
- Make `m_daemonMutex` `mutable` so it can be locked from the `const` `stopWatcher()`.

### User-visible effect

- Daemon console actually displays monerod output again
- `daemonStopped()` now fires when monerod exits, so the UI can react to crashes instead of relying solely on RPC polling.
- Stopping monerod/p2pool from the GUI no longer kills unrelated instances on the same machine.

### Compatibility notes

- On Windows, the previous code already omitted `--detach` but still used the static `startDetached(program, args)` overload, so the signals were broken there too. This change fixes both platforms uniformly.
- If the GUI is hard-killed (SIGKILL), monerod is orphaned to init on Unix (`QProcess` does not auto-kill children on Unix), matching the previous behavior. On Windows, a hard GUI crash now terminates monerod (previously the `taskkill` in the destructor path never ran and monerod leaked). Graceful shutdown still goes through the RPC `exit` command exactly as before.
- No QML or public API changes; `DaemonManager` and `P2PoolManager` invokable signatures are unchanged.
