# Original Description
Since recently, the Qt project build itself relies on files autogenerated by monero build (`monero/src/wallet/trezor_link_flags.txt`). This breaks Qt project builds invoked outside of the upstream build script (makes the Qt project build no longer self-sufficient like it used to be).

The Qt project build is invoked outside of upstream build script by the distribution package recipes (monero-wallet-qt on Arch), because it should not need to build monero just to build the wallet (because monero is in a separate package). So, this design breaks distro package builds.

Does the dependency list really need to be generated by monero? It should be sufficient to add matching feature flags to the Qt project that can be enabled on the build command line, and have the build script (and the package recipe) simply pass consistent feature settings to both monero build and wallet Qt project build; and have the Qt project append linked libraries based on the feature flag. This hiding/abstraction of the dependency list for a given feature from the wallet Qt project causes problems without a clear benefit. Agreed? I can try to PR.

My current [workaround](https://aur.archlinux.org/cgit/aur.git/commit/?h=monero-wallet-qt-git&id=3b20a5c686ddbe8b7a5f4f0d6c3cbd08125ca5aa) is to maintain patches in the package recipe that add dependencies to the link line in .pro file: -lprotobuf and -lusb-1.0. Would be nice to go back to no patches needed.
