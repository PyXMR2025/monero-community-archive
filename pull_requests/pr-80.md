# Original Description
## Part 8
https://github.com/Cuprate/cuprate/pull/79 <- Previous Next -> https://github.com/Cuprate/cuprate/pull/81

- Removes the `sanakirja` backend, documents why
- Adds [`redb`](https://docs.rs/redb) and implements the `cuprate_database` traits (just type checks, no body)
- Actually implements `redb::Error` -> `cuprate_database::RuntimeError`

## Traits
The core traits were also changed to account for `redb` abstracting RW operations at the table level (with an associated TX lifetime) compared to `heed` abstracting that at the TX level.

- `trait Database` -> `trait DatabaseRo`, `trait DatabaseRw: DatabaseRo`
- `Env::open_database()` -> `Env::open_db_r{o,w}()`
-  Added `HeedTableR{o,w}` that mimics `redb`'s [`ReadOnlyTable`](https://docs.rs/redb/1.5.0/redb/struct.ReadOnlyTable.html) & [`Table`](https://docs.rs/redb/1.5.0/redb/struct.Table.html)

Now, instead of `trait Database` taking a TX as input every function, the struct that actually implements `trait DatabaseR{o,w}` will need to hold onto the transaction somehow. This is [already the case](https://docs.rs/redb/1.5.0/src/redb/table.rs.html#61) in `redb`'s table types, and is mimicked in our custom `HeedTableR{o,w}` types.

We should still be able to reuse TX's since `HeedTableR{o,w}` only take references:
```rust
let env: Env = todo!();
let mut tx: TxRw = env.tx_rw();

{
    let db: DatabaseRw = env.open_db_rw::<Table>(&mut tx);
    /* do db stuff */
}

{
    // We still have access to `tx`.
    let db2: DatabaseRw = env.open_db_rw::<Table2>(&mut tx);
    /* do db stuff */
}

// Re-used the `tx` for 2 table operations :)
tx.commit();
```

## `SyncMode`
For now, all backends will only support:
- `SyncMode::Safe`
- `SyncMode::Async` (added in this PR)
- `SyncMode::Fast`

as they are easy to implement (just set once in `Env::open`).

Dynamic syncs are more difficulty. `redb` also abstracts it at the TX level, where as `heed` abstracts it at the `Env` level. `heed` does not expose `mdb_env_set_flags` which is needed to change sync modes dynamically (without re-opening the `Env`), so dynamic syncs will be implemented in some later PR.
