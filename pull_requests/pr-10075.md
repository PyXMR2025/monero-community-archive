# Original Description
#10064

- reduce query frequency from 3600s -> 120s
- reduce ban time from 86400s to 300s
- increase requirement for dns servers to agree, from  `1 greater than 1/2` to `1 greater than 2/3`
- add remainder of moneropulse domains

Note: there is an issue unrelated to the PR that can leave nodes in a bad state
1. Node is reorged
2. Node receives a checkpoint that references the now orphaned chain
3. Node refuses to switch to the checkpointed chain, causing the node to get stuck. It will reject new blocks that dont match the checkpoints as well as rejecting the, currently orphaned, checkpointed chain

i believe its a logic issue with the `if` condition on L1899-2124 of src/cryptonote_core/blockchain.cpp [src](https://github.com/monero-project/monero/blob/8e9ab9677f90492bca3c7555a246f2a8677bd570/src/cryptonote_core/blockchain.cpp#L2089)

If there is a checkpoint, it _should_ reorg onto the checkpointed chain, not complain about it being orphaned and reject it.

The answer to the FIXME on [L2089](https://github.com/monero-project/monero/blob/8e9ab9677f90492bca3c7555a246f2a8677bd570/src/cryptonote_core/blockchain.cpp#L2089) is "yes".
