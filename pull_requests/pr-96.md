## Part 16
https://github.com/Cuprate/cuprate/pull/95 <- Previous Next -> https://github.com/Cuprate/cuprate/pull/97

This add various functions to `trait DatabaseR{o,w}` that are supported by both backends `heed` & `redb` - more specifically, APIs that are `BTreeMap`-like are added since both are based on B-trees.

All methods were added to the current tests in `src/backend/tests.rs`.

```rust
fn iter(
    &self,
) -> Result<impl Iterator<Item = Result<(T::Key, T::Value), RuntimeError>> + '_, RuntimeError>;

fn keys(
    &self,
) -> Result<impl Iterator<Item = Result<T::Key, RuntimeError>> + '_, RuntimeError>;  

fn values(
    &self,
) -> Result<impl Iterator<Item = Result<T::Value, RuntimeError>> + '_, RuntimeError>; 

fn contains(&self, key: &T::Key) -> Result<bool, RuntimeError>;

fn len(&self) -> Result<u64, RuntimeError>; 

fn first(&self) -> Result<(T::Key, T::Value), RuntimeError>; 

fn last(&self) -> Result<(T::Key, T::Value), RuntimeError>; 

fn is_empty(&self) -> Result<bool, RuntimeError>; 
```

## hinto-janai | 2024-03-27T18:15:33+00:00
The main reason is because `?` no longer works for callers, aka, any non-`service` user.

You can no longer use `?` because the error might be a non-existent key (not really a hard error) or an _actual_ database error, so all callers must:
```rust
fn get_block(db: &Database) -> Result<Option<Block>, Error> {
    match db.get(&key) {
        Ok(v) => Ok(Some(v)),
        Err(RuntimeError::KeyNotFound) => Ok(None),
        Err(e) => Err(e),
    }
}
```
`Option` is probably used in most APIs because it represents the problem more accurately and allows `?`:
```rust
fn get_block(db: &Database) -> Result<Option<Block>, Error> {
    Ok(db.get(&key)?)
}
```
With `Option`, yes `cuprate_database::service` will have to map it but taking away `?` from non-`service` users and forcing `match` at all call sites is also quite bad. Although, I guess it does make usage within Cuprate easier.

## hinto-janai | 2024-03-27T18:59:18+00:00
Actually, this is hard to impl actually since `redb` doesn't allow deleting tables that are open, [it returns `TableAlreadyOpen`](https://docs.rs/redb/2.0.0/src/redb/transactions.rs.html#380-382).

If our custom `RedbTable` itself holds onto the table we want to delete, we have to do some swapping like so:

1. swap our `self.db` table with a temporary table
2. delete the target table
3. create the target table, set `self.db` to it again
4. delete the temporary table

Only issue is that this is not `Drop`-safe, as in if we early return or panic, our `self.db` is set to the invalid temporary table. Creating a special struct and doing `fn drop() { /* reset self.db */ }` is not good either since that can error too.

I think there are only 2 safe alternatives:
1. Use `redb::Table::retain(self, false)` which hopefully compiles down efficiently since the predicate is known
2. Just remove `DatabaseRw::clear()` from this PR
