## vtnerd | 2018-04-11T18:35:45+00:00
I think it would be more appropriate to declare/define a function local struct that contains these static values, then declare a static instance of that struct. The reasoning is that neither `boost::mutex` nor `std::vector` is marked as `constexpr`, so both may require runtime initialization on first access. Moving them into a single struct will help the compiler "batch" these atomic-synchronization events into a single event, whereas the compiler hits a memory barrier with that lock construction right now.

## vtnerd | 2018-04-12T16:32:04+00:00
The current implementation doesn't use the cache unless the start and end values are exactly the same. It should be possible to use the cache for any subset within the cached range, which is what my code snippet was attempting to do. Perhaps that isn't possible in the wallet implementation currently? It would help still help with any _outside_ implementations using that call (which seem rare fwiw).

## vtnerd | 2018-04-12T16:38:20+00:00
Yes this is a micro-optimization. Each **function-local** static variable has to be initialized at compile-time, or if not possible, synchronized on first use at run-time. `boost::mutex` and `std::vector` are not marked as `constexpr`, and therefore are more likely to require separate runtime initialization (double-check lock), particularly because the current code has a lock acquisition between the statics. The compiler could probably legally batch the static creation double-check locking, but probably not with that lock in between. Putting them into a struct just makes it easy to declare that all of these need to be/can be initialized at once.

Also `std::mutex` is marked as `constexpr`, which is one of its advantages over boost right now.
