# Original Description
## Description

A while ago, this PR was merged https://github.com/monero-project/monero/pull/8941 (@moneromooo-monero)

The consequence of this was that the monero-wallet-rpc would hang on auto-refreshes. This is an example sequence of events that causes the hang:

- Restore wallet from a low height (e.g. 2,500,000)
- Refresh event begins (kicked off every second)
- More than REFRESH_INFICATIVE_BLOCK_CHUNK_SIZE is fetched, m_last_auto_refresh_time is not updated; furthermore,
- Over 1000ms have passed since the beginning of the refresh event.
- Next refresh cycle is immediately kicked off.

This continuous refresh event, without stop, blocks the wallet-rpc from responding to other events.

 ### How to recreate issue on Master

1. Create a wallet, either programatically (monero-wallet-rpc) or manually (monero-wallet-cli)
2. Get the address, private spend key, and private view key.
3. Call [generate_from_keys](https://docs.getmonero.org/rpc-library/wallet-rpc/#generate_from_keys) with a restore_height of `2500000`
4. Try to hit the RPC with `get_version` (e.g. `curl -X POST http://127.0.0.1:18082/json_rpc -d '{"jsonrpc":"2.0","id":"0","method":"get_version"}' -H 'Content-Type: application/json'`), you'll notice it just hangs (make sure you have the correct host and port, I bound my monero-wallet-rpc to 18082).

## Change

This PR introduces a between 200 - 300ms evaluation period whereby the network thread is free, if we've non-stop processed blocks for over m_auto_refresh_period. This gives us a 'breathing room' period by which we can handle other rpc-requests.

Observe the timestamps below to understand the sequence of events. The image below is included when the evaluation period was 3 seconds. It's been lowered to the interval above (200 - 300ms).

## Tzadiko | 2025-03-28T23:20:11+00:00
After speaking to Nahuhh (ofrnxmr) in Element, we've settled on ~[200ms to 300ms] worth of waiting after the `m_auto_refresh_period`.

There’s no way to tell if a request is pending. The code just isn’t written that way. The refresh hogs the CPU, the only thing we can do is give it a temporary break. You can reduce the temporary break as a percentage of the total time spent by increasing the m_auto_refresh_time from the default 20 seconds to a larger number.
