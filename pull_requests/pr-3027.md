# Original Description
I was inspired by a reddit post and came up with a new kind of cryptographic proof that proves your partial reserve to an auditor (https://www.reddit.com/r/Monero/comments/7mkoi8/random_question/druqe05/), called *reserve proof*. The basic idea is to disclose only a fixed set of your unspent outputs with signed key images (and to **NOT** hand over your view secret key). This subset can be either the entirety of your available outputs across all subaddress accounts, or a set of outputs within the currently selected account that can cover the requested amount with the smallest set size.

Here's the command synopsis:
```
Command usage: 
  get_reserve_proof (all|<amount>) [<message>]

Command description: 
  Generate a signature proving that you own at least this much, optionally with a challenge string <message>.
  If 'all' is specified, you prove the entire sum of all of your existing accounts' balances.
  Otherwise, you prove the reserve of the smallest possible amount above <amount> available in your current account.


Command usage: 
  check_reserve_proof <address> <signature_file> [<message>]

Command description: 
  Check a signature proving that the owner of <address> holds at least this much, optionally with a challenge string <message>.
```

## stoffu | 2018-01-02T13:47:47+00:00
Yes, it's intentional. Let me summarize these different proof schemes which are the following three:

1. Tx proof,  which proves: "I know the tx secret key of this `<txid>` where this much was sent to that `<address>`".
2. Spend proof, which proves: "I am the author of this `<txid>` hence I can recreate a new set of ring signatures consisting of the exact same set of decoy outputs (I can do so because I know the secret key of the real spent output of each ring)."
3. Reserve proof, which proves: "I know the secret key of this output key whose key image is that. Also its Pedersen Commitment commits to this amount."

As you can see, the address is part of the proof for 1, but not for 2 & 3. Hence 2 & 3 don't take address as input.
