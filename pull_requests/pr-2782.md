# Original Description
These cases were generating errors that are quite misleading. They happen easily when the user confuses the order in the commands and swap the ring_size and the amount.

```
> transfer 5.4 <address>
Error: failed to parse address
```
... while it has obviously nothing to do with the address being wrong.

```
> transfer 5.4 <address> 5
Error: payment id has invalid format, expected 16 or 64 character hex string: 5
```
... while it has nothing to do with the payment ID being wrong -- it's not even provided.

The same was true when more arguments are provided (for instance priority) and for all similar calls:
* locked_transfer
* transfer
* transfer_original
* sweep_all
* sweep_below

In all these variations, command such as the ones above now produce the following error:
"ring size must be an integer >= 3"
