# Original Description
When we got blocks to be large, xmrig was reporting ~2815 txs for the job. Mining the block results in the node segfaulting.

Triggered multiple segfaults on a both 1TB ram + 256 thread, and 16GB ram + 8 thread devices. (it doesnt seem to have anything to do with resources)

Using monerod to mine the block doesnt seem to cause trigger a segfault.

Using xmrig for smaller blocks also didnt seem to be an issue either.

At this point the TCP server is waiting for the `on_write` handler to be executed, but that handler is waiting on the intermediate `async_write`s to complete, which is waiting for the remote end to read data (which it never does), and thus fills up a permanent slot in our connection list.

I don't think there is a cleaner solution other than forcibly closing the socket, as waiting on the remote side to read data is an exhaustion attack.

Making clear my findings: the root issue is that the server can technically close a socket *while* a write is still getting processed, causing the segfault (socket closes, boost attempts a write on destructed socket concurrently).

Because of the fact you're pointing out that `async_write` can get blocked by a malicious client, I don't even think bringing back `wait_write` would work as a viable solution to this issue even with a timeout. Implementing a timeout won't fully solve the issue because we have to guarantee that the write is not getting processed at the same time as socket closure; however, with a timeout, it would be possible for that to happen right around timeout expiry i.e. segfault still possible. Thus, AFAICT the only viable solution to that root issue that I see is to make sure the server closes the socket in the read/write `m_strand`.

Thus, in https://github.com/monero-project/monero/pull/10278 , I solved the issue by making sure to only ever close the socket in `m_strand`, and as a bonus: when the client includes the "Connection: close" header, I make sure to send a complete response to the client before closing the connection. In the end, I left those `wait_write`'s commented out. The solution is net ~60 new lines of code and not too complicated imo.
