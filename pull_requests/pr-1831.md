# Original Description
Explains the decoy selection bugs and their impact, and recommends users update to v0.17.2.3 as soon as able.

### On bug 1 (incorrectly applying the distribution of spending patterns)

Note the following section:

> Users should also be aware that the implications of the first bug are not as severe as previously reported. It was [widely publicized](https://decrypt.co/76938/monero-developers-disclose-significant-bug-privacy-algorithm) that as a result of the first bug, some very newly spent outputs observed on chain were guaranteed identifiable as real outputs spent in a transaction, however, *this is not actually the case*. Since publication, we learned that [MyMonero](https://mymonero.com/) — a popular light wallet — has been using a [separate implementation](https://github.com/vtnerd/monero-lws/blob/faa51780f3f8e6c5c0c4235499b95c246e074f29/src/util/gamma_picker.cpp) of the decoy selection algorithm that *did not have the bug*. Therefore, MyMonero users could have feasibly constructed transactions that selected newly spendable outputs as decoys, which means newly spent outputs observed on chain were not guaranteed identifiable as real outputs.

In my initial draft, I included the following sentence in a new pargraph:

> Note that MyMonero is a light wallet that stores transaction data for its users on their servers, which means that MyMonero may still be able to use the transaction data stored on their servers to better identify some very new real outputs on chain as a result of the bug.

I received conflicting feedback on including/excluding this, and opted to exclude it because I don't know if it's 100% true for 2 reasons:

1. MyMonero is using the `monero-lws` implementation, which does not have the bug. I don't know how many `monero-lws` users there are out there, and don't believe there is a way of finding out with certainty. Therefore the existence of `monero-lws` provides plausible deniability for early spents observed on chain, even from MyMonero using their transaction data to try to infer real output data on chain.
2. Perhaps there is also some other implementation out there we don't know about that is selecting younger outputs.

It seemed too confusing to fit all that in there, so I left it out.

### On bug 2 (integer truncation)

I kept the focus on the divide by 0 aspect of the bug. I didn't talk about how it causes the algorithm to select older outputs in higher frequency, and younger outputs in lower frequency, which has privacy implications as well. That aspect of the bug was not patched in v0.17.2.3, since it wasn't clear if it was safe to do (on grounds of transaction uniformity, and on grounds of it potentially selecting a "beneficial" number of older outputs). However, the privacy implications of selecting fewer younger outputs as a result of the integer truncation bug are partially mitigated by the solution to bug 1, where outputs that are roughtly <50 blocks old outputs are re-selected in higher frequency. So, because of this, it seems justified not to go into detail on the integer truncation bug's privacy implications, because of how confusing it would get.

## erciccione | 2021-09-16T08:56:18+00:00
```suggestion
When constructing a new @transaction, a Monero wallet references a past output a user received in a prior transaction, and uses it as an input to the new transaction. Today, the wallet mixes this output with a set of 10 decoy outputs selected from other people's transactions from across the blockchain (@ring-signature). Thus, an observer cannot tell which among the 11 total outputs is the real one a user spends in a transaction, obfuscating the link from the user's new transaction to their prior transaction. The decoy selection algorithm specifically handles the selection process for choosing which decoys to mix real outputs with when constructing a transaction.
```
