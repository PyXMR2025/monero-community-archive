## stoffu | 2017-08-13T13:48:47+00:00
I think `multisig_k` should be cleared here in order to prevent an attacker from stealing the spend secret key in the following way:

Consider a scenario with a 3/3 multisig wallet, where Alice, Bob and Charlie are the participants. Alice and Charlie are colluding together to steal the spend secret key from Bob. The shared view secret key is `a` while the spend secret keys of Alice, Bob and Charlie are `b_Alice`, `b_Bob` and `b_Charlie`, respectively.

### 1. Alice creates a transaction and passes it to Bob.

At this point, the partial signature scalar amounts to:

```
s_step1 = k_Alice - c * (Hs(a*R) + b_Alice)
```

### 2. Bob signs it with his spend secret key and passes it to Charlie.

At this point, the partial signature scalar amounts to:

```
s_step2 = k_Alice + k_Bob - c * (Hs(a*R) + b_Alice + b_Bob)
```

### 3. Charlie shares the partially signed transaction with Alice.

At this point, Alice and Charlie can know Bob's contribution to the signature scalar by taking the difference:

```
s_diff = s_step2 - s_step1
       = k_Bob - c * b_Bob
```

### 4. Alice creates another transaction and passes it to Bob, saying "Sorry, there was a mistake in the transaction I made previously, so please sign again this new transaction."

### 5. Bob signs the new transaction and passes it to Charlie.

### 6. Charlie shares the partially signed new transaction with Alice.

At this point, Alice and Charlie do exactly the same computation as above, and obtain another instance of Bob's contribution to the signature scalar using his same `k_Bob` and a different `c_new`:

```
s_diff_new = k_Bob - c_new * b_Bob
```

### 7. Alice and Charlie can extract Bob's spend secret key as
```
b_Bob = (s_diff_new - s_diff) / (c_new - c)
```

To prevent this attack, the `multisig_k` should never be reused, and it should be cleared every time a participant signs a transaction (either via `transfer` or `sign_multisig`). If a partially signed transaction is to be cancelled, the export/import of multisig info should be performed again.

## stoffu | 2017-11-28T23:42:29+00:00
@JollyMort 
> Why not just roll a new random key when calling `prepare_multisig`.

> So, if we roll a new seed, present it to the user, have him confirm that he has written it down before letting him proceed then that ought to be ok, no? 

My idea is to not generate a new wallet key for each multisig wallet creation, but rather keep using the current normal wallet as-is, and generate a new separate multisig-only wallet _**with the same initial spend secret key (seed)**_. After completing the preparation steps of key exchange with other participants, such a multisig wallet gets its own address and partial secret key(s), which is stored in a separate file (so the original wallet can still be used as a normal full wallet).

> And the purpose of the salt is not clear to me if it's public and constant.

> In the MRL paper, the salt is supposed to be like a passphrase chosen by the user, different for each set-up, but that doesn't seem user-friendly to me TBH.

I haven't read the MRL paper yet, but AIUI the idea is to allow each participant to hide the connection to the original spend public key `B=b*G` through blinding: `b' = Hs(b || salt); B' = b'*G`. In this sense, salt is just a public constant, just like how the subaddress key is computed using another salt: `D = Hs(a || index || salt)*G + B`.

IMHO, though, the benefit of salting (i.e. hide the original wallet address from other participants) doesn't seem too huge to me.
