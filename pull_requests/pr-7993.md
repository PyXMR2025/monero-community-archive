# Original Description
#7798 is a blocker for this PR

Proposing to decrease the recent spend window from 50 to 15, so that the decoy selection algorithm will select a marginally higher number of outputs from the first 15 spendable blocks, rather than from the first 50 spendable blocks. This isn't a major change, only a marginal one. Note that the analysis below uses data from *before* the anomalous spike on July 21st.

### Overview

The "recent spend window" is meant to capture the spending patterns of Monero users who spent outputs <10 blocks old back when the lock time wasn't enforced by consensus -- the gamma distribution used in the decoy selection algorithm was apparently fit [using that data](https://github.com/maltemoeser/moneropaper/blob/master/Monero%20Analysis.ipynb). The idea is that if the gamma suggests an output that is <10 blocks old today (i.e. an unspendable output), it should re-select an output from a recent window to more closely mirror expected spending patterns when the gamma was fit, rather than simply re-draw an output from the entire chain using the gamma. Re-drawing from the gamma in the latter case would ignore the spending patterns of users who spent in the early window. Note that this re-draw occurs relatively infrequently, and so this is not a major change to the algorithm. The aim of this change is to fit observed and expected spending patterns *marginally* better.

The recent spend window's magic number of 50 was chosen assuming integer truncation would not be patched for v0.17.2.3 (reasoning [here](https://github.com/monero-project/monero/pull/7821#issuecomment-898258800)). However, patching integer truncation in #7798 causes the client to select more outputs in the 10-100 block range, and therefore the benefit of choosing a larger recent spend window of 50 has less merit.

Based on the above, I originally went with 10. @SamsungGalaxyPlayer suggested 10 or 15 as well in the latest [dev meeting](https://github.com/monero-project/meta/issues/614), and I recall support for something around 15 in dev discussions a while back. I agree 15 would be a good choice too. If you assume the wallet2/monero-lws implementations were the only ones used over these epochs, it would be ok for the updated parameter selection to be "above" the blue line, because the gap *below* blue would indicate a proportionally larger number of reals that could use an even larger number of decoys to cover the reals. With this logic, 10 or 15 seem well-suited to cover the apparent early triangular gap from blue to orange observed in all epochs.
