# Original Description
Unlike my earlier PR #4134 back in July with preview code this here is the completed initial version of the MMS (Multisig Messaging System) that I would like to submit for review and later inclusion into the Monero codebase, hopefully ready for going into service with the next hardfork in Spring.

The system was beta-tested extensively, and its functionality was improved and refined over several iterations already.

To make reviewing this considerable amount of new and changed code easier I have written the following text with high-level information about the implementation of the MMS: https://rbrunner7.github.io/mms_code

There you also find links to explanations what this MMS is about in the first place, and instructions how to get the MMS running for testing purposes.

There currently is only a weak dependency on a single other PR, PR #4845. The bug fixed there only strikes with multisig with more than 3 coalition members however, e.g. 2/4.

The auto-config mechanism works like that: You receive from me an *auto-config token* e.g. by SMS, phone, or other "reasonably secure" channel; you type that in; the MMS uses the token as seed to derive a single-use Bitmessage address and sends a special auto-config data message to that address, containing your Monero address and your "real" Bitmessage address that you want to use for transacting with me. Because my wallet generated the token it knows on which single-use Bitmessage your data will arrive. Note that auto-config data is encrypted, with a key pair that was also derived from the auto-config token.

If you make a mistake when typing in the token, the MMS would send your auto-config data to a completely different Bitmessage address. Result: My wallet does not receive anything.

I think a primitive checksum should already do the job.
