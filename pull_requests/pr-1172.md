```c++
  template<typename T>
  struct rpc_handler
  {
    static std::pair<const char*, rpc_handler<T>> make()
    {
      return std::make_pair(T::name, rpc_handler<T>{});
    }
    std::string operator()(DaemonHandler& handler, const rapidjson::Value& in) const
    {
      return handler.handle(T::fromJson(in)).toJson();
    }
  };

  std::map<const char*, std::function<std::string(DaemonHandler&, const rapidjson::Value&)>> methods;
  methods.insert(rpc_handler<GetBlocksFast>::make());
```

Storing a `const char*` in the `std::map` is a little iffy, but should work in this situation (its preventing a temporary std::string allocation from the rapidjson lookup). Or `std::unordered_map` too. Removes the linear search (which has few elements, luckily).

## vtnerd | 2016-12-05T22:36:07+00:00
Should these `fromJson` methods be static, and return an instance of the message class? There is little benefit to making the function virtual; a concrete class of this type has to be created before invoking the fromJson method anyway.

This code is inefficient when going to JSON in particular. The request/response struct IS a DOM, but these functions are copying and allocating memory for a JSON DOM first. Its (relatively) easy to serialize JSON from a struct if `boost::spirit::karma` is leveraged. Ignore the horrors from epee for a second here - I wrote this recently and I can post it here if you want to look at it for an example. C++11 variadics + boost::fusion::for_each make it easy to write out an object without any weird state tracking that epee or cereal does for this purpose.

Deserializing without an intermediary DOM is a little tricky because object members can be given in any order. Anyway if the signature were changed, _how_ the objects were being converted can be changed at any time. The only negative is if the code wanted to use the JSON DOM directly, but that never appears to be the case.
