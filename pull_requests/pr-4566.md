# Original Description
This PR implements a [long requested community feature](https://bounties.monero.social/posts/68/6-026m-statistical-mining-dashboard-for-p2pool-in-monero-gui-wallet-2m) intended to educate newcomers on mining concepts and expose them to p2pool.

This adds:
- **A New Dashboard View**: Accessible from "Advanced > Mining" when "P2Pool" is selected as the mining mode. Implements buttons with transitions and theme compatibility, as well as a user interface and experience matching the rest of the program.
- **Educational Resources**: An on-demand explanation for each dashboard field to help educate newcomers on p2pool and mining terminology and concepts.
- **Refined Data Visualization**: Friendly readouts for hashrate, with rolling averages, and mining effort, alongside an "Advanced" section for those who wish to view the raw API data.
- **Synchronization Gating**: Waits until the side and blockchains are synced to address some consistency issues. The initial synchronization takes some time, and the reporting from the API is a bit eager since we're using P2Pools built-in miner which itself doesn't wait.

This is intended to supersede #4182 which was a partial implementation lacking most of the important points above.

## thomasbuilds | 2026-04-26T18:13:35+00:00
**Issue: `getStatus()` has hidden side effects on EMA / effort state**

`getStatus()` is called on its own cadence by `Mining.qml`, but it now invokes `fetchLocal()`, which mutates `m_hashrate_ema15m/1h/24h`, `m_effort`, `m_round_hashes` and `m_shares_found`. However, `m_lastUpdate` is only assigned in `update()` never in `fetchLocal()`.

Consequences:

- While the user sits on the Mining page, every `getStatus()` tick advances the EMAs but `m_lastUpdate` keeps growing stale, so `secs = m_lastUpdate.secsTo(now)` blows up and `alpha → 1.0` each new sample fully replaces the EMA, defeating the smoothing.
- When the dashboard is then opened, `update()` sees state already corrupted by `getStatus()` calls.
- Two callers (Mining tick + Dashboard 5 s timer) interleave and double-count samples within a window.

**Suggested fix:** split `fetchLocal()` into a *pure* formatter (no member writes) used by `getStatus()`, and keep the stateful EMA/effort step inside `update()` only. Roughly:

```cpp
// pure: just reads files and formats; no member writes
QVariantMap fetchLocalSnapshot() const;

// stateful: snapshot + advance EMAs + set m_lastUpdate
void recomputeAggregates(const QVariantMap& local, const QVariantMap& pool);
```

Then `getStatus()` calls `fetchLocalSnapshot()` and `update()` calls both.

## thomasbuilds | 2026-04-26T18:15:26+00:00
**Issue: holding `const T&` to `P2PoolManager`'s members is a footgun**

`P2PoolStatsProvider` keeps `const bool& m_started` and `const QString& m_p2poolPath` referencing fields owned by `P2PoolManager`. It works today only because the parent owns the provider via `unique_ptr` and outlives it. Any future refactor that:

- moves the provider onto a worker thread,
- copies/moves it,
- or reorders member destruction in `P2PoolManager`,

will silently produce UB the references will dangle or race with writes in `start()`/`exit()` with no compiler warning.

**Suggested fix:** take a back-pointer to the parent and add small accessors, or pass values + setters. For example:

```cpp
explicit P2PoolStatsProvider(P2PoolManager* parent);
// ...
private:
    P2PoolManager* m_manager;  // non-owning, parent guarantees lifetime
    bool started() const { return m_manager->isStarted(); }
    QString poolPath() const { return m_manager->p2poolPath(); }
```

This also keeps the threading story explicit if you ever move stats fetching off the GUI thread.

## thomasbuilds | 2026-04-26T18:18:07+00:00
**Issue: each `update()` re-reads & re-parses the same JSON files ~10×**

Within a single tick:

- `fetchLocal()` reads `local/miner` + `pool/stats`, then calls `isSidechainReady()` which reads them **again**.
- `fetchPool()` reads `pool/stats` + `local/miner` and again calls `isSidechainReady()`.
- `fetchNetwork()` reads `network/stats` and `isNetworkReady()` reads it again.
- `fetchRaw()` reads everything one more time.

That's roughly 10 file opens + JSON parses every 5 s, all redundant.

**Suggested fix:** parse each file once at the top of `update()` and pass the maps down:

```cpp
void P2PoolStatsProvider::update()
{
    const QVariantMap localRaw   = fetch("local/miner");
    const QVariantMap poolRaw    = fetch("pool/stats");
    const QVariantMap poolStats  = poolRaw["pool_statistics"].toMap();
    const QVariantMap networkRaw = fetch("network/stats");

    const bool sidechainReady = poolStats["sidechainHeight"].toULongLong() > 1000
                              && localRaw["block_reward_share_percent"].toInt() != 100;
    const bool networkReady   = networkRaw["height"].toULongLong() > 0;

    auto local   = buildLocal(localRaw, poolStats, sidechainReady);
    auto pool    = buildPool(poolStats, localRaw, sidechainReady);
    auto network = buildNetwork(networkRaw, networkReady);
    auto raw     = buildRaw(localRaw, poolRaw, networkRaw);

    m_lastUpdate = QDateTime::currentDateTime();
    emit p2poolUpdateStats(local, pool, network, raw);
}
```
